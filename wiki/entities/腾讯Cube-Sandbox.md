# 腾讯Cube-Sandbox

**腾讯Cube-Sandbox** 是一种高性能沙盒执行环境，专为大规模 [[Agent]] 并发执行场景设计。在 [[蜂群思维：多Agent协作架构的“版本答案”]] 报告中，它被定位为实现“脑手解耦”的关键基础设施组件——即作为 Agent 的“双手”，承载所有代码执行、环境操作和工具调用，而将模型决策与上下文管理剥离到上层架构中。

## 核心特性

### 毫秒级启动与销毁
Cube-Sandbox 的核心能力在于其极快的生命周期管理。与传统 Docker 容器需要数秒甚至数十秒的启动时间不同，Cube-Sandbox 能够在毫秒级别完成环境的创建与销毁。这一特性使其非常适合 Agent 的“无状态化”执行模式：每个任务或任务步骤都可以在一个全新的、干净的沙盒中执行，任务完成后立即销毁，不留任何残留状态。

### 环境隔离与故障隔离
每个 Cube-Sandbox 实例都是完全隔离的执行环境，拥有独立的文件系统、网络栈和进程空间。这种隔离机制确保了：
- **故障隔离**：单个沙盒内的代码崩溃、内存泄漏或恶意操作不会影响其他沙盒或宿主系统。
- **环境纯净**：每次执行都从预定义的基线镜像启动，避免了“环境配置漂移”导致的难以复现的 Bug。
- **安全执行**：用户提交的代码在受限环境中运行，无法访问宿主系统的敏感资源。

### 与 Agent 架构的深度集成
在 [[Managed Agents]] 架构中，Cube-Sandbox 作为执行层，与上层的 Session 管理和编排器（Coordinator）紧密配合：
- **Session 绑定**：每个 [[Session]] 可以绑定一个或多个 Cube-Sandbox 实例，用于执行该 Session 内的代码任务。
- **编排器调度**：编排器在拆解复杂任务后，会为每个子任务分配一个独立的 Cube-Sandbox，确保子任务之间互不干扰。
- **状态无状态化**：Agent 的“大脑”（模型）不直接与沙盒交互，而是通过 Session Store 读取上下文，再通过编排器向沙盒下发指令。沙盒本身不存储任何持久化状态，所有执行结果通过标准输出/错误流返回。

## 技术细节

### 镜像管理
Cube-Sandbox 支持预构建的镜像模板，这些模板预装了常用的开发工具、库和运行时环境（如 Python、Node.js、Go 等）。开发者可以根据任务需求选择或自定义镜像，实现“即用即走”的执行环境。

### 资源配额
每个沙盒实例都有严格的资源配额限制，包括 CPU 时间、内存上限、磁盘空间和网络带宽。这防止了单个任务过度消耗集群资源，保障了整体系统的稳定性。

### 日志与审计
所有在沙盒内执行的命令、产生的输出以及资源消耗都会被完整记录，形成可审计的执行日志。这对于调试、性能优化和安全审查至关重要。

## 在蜂群架构中的角色

在 [[蜂群思维：多Agent协作架构的“版本答案”]] 所描述的架构中，Cube-Sandbox 扮演了“牛马”的角色——它被设计为可随时创建、使用和销毁的廉价计算资源，而非需要精心维护的“宠物”。这种设计哲学使得 Agent 系统能够：
- **水平扩展**：通过增加沙盒实例数量，轻松应对并发任务高峰。
- **快速恢复**：当某个沙盒内的执行出现异常时，编排器可以立即销毁该沙盒并创建一个新的沙盒重试任务，整个过程对用户透明。
- **成本优化**：沙盒仅在任务执行期间存在，空闲时自动释放，避免了长期占用计算资源带来的浪费。
