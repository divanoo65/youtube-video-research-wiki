# Anthropic

**Anthropic** 是一家专注于人工智能安全与研究的公司，其在2026年4月（根据报告引用）提出的 **Managed Agents（托管Agent）** 架构，被认为是多Agent协作领域的关键突破。该架构通过系统性解耦决策层与执行层，解决了传统单一Agent在长程任务中容易崩溃、记忆污染及环境配置冲突等核心痛点，推动了AI Agent从“个人工具”向“企业基础设施”的技术范式转移。

## 核心思想：脑手解耦与无状态化

Managed Agents 架构的核心思想是“脑手分离”。报告指出，Anthropic在其博客中强调：“你要扩大Agent的规模，就要解耦‘脑’和‘手’。” 这种解构使得模型的规划能力（脑）与代码执行环境（手）彻底分离。

- **从“宠物”到“牛马”**：报告引用了一个形象的比喻：“要把Agent当牛马去用，而不是当宠物去养。” 在传统架构中，Agent如同需要精心维护的宠物，其环境配置与记忆高度耦合，一旦出错则系统崩溃。而在Anthropic的架构中，Agent被视为无状态的生产工具。
- **沙盒执行**：每一项执行任务都在独立的沙盒环境（如高性能的Cube Sandbox）中运行。如果任务出错或环境被污染，系统会直接销毁该沙盒并重新拉起一个干净的环境，从而确保了系统的健壮性与故障隔离。

## 四层解耦架构

根据对Anthropic SDK及相关实践的剖析，Managed Agents 架构具体表现为四个层级的解耦：

1.  **Session 与 Agent 的实体分离**：在该架构中，[[Agent]]与[[Session]]被视为两个完全不同的实体。
    - **Agent（静态模板）**：定义Agent的身分、所使用的工具、模型以及系统提示词（System Prompt）。它是一个不存储动态数据的配置模板。
    - **Session（动态执行）**：承载对话线程、记忆挂载和文件操作。报告用一句话精辟地总结了这种关系：“Session负责我在干什么，Agent负责我是谁。” 同一个Agent模板可以同时启动多个独立的Session，互不干扰。

2.  **编排者（Coordinator）模式**：通过分析SDK结构，发现了`coordinator`字段，它代表一种“不干活的指挥官”角色。
    - **职责**：负责拆解复杂任务，并派发指令给下属Agent。一个编排者名下可管理多达20个拥有独立Prompt和MCP（Model Context Protocol）服务器的Agent。
    - **解决问题**：该模式解决了单一Agent在长程任务中因上下文不足而“变笨”的问题。通过调度不同的Agent（如编码、审核、测试）分工协作，每个节点都能保持上下文的清晰。

3.  **上下文解耦（Session Store）**：这是架构中最具突破性的部分。通过将Context从本地转移到云端（如[[Redis]]缓存），实现了Agent的彻底无状态化。
    - **功能**：Session可以作为“记忆树”进行Fork（分支）、回滚、克隆或恢复。报告提到：“作者在发现Session Store实现上下文云端化后，对整套解耦架构体系的最终定性是：‘这不就是一个Agent的操作系统。’”。

## 影响与实践意义

- **多模型组合与Token效率**：通过任务图（Task Graph）预定义规则，可以实现大小模型的异构组合。多个节点可以并发执行搜索、编码或写作任务，并根据任务阶段灵活调用不同性能的模型，从而优化Token成本。
- **资源变革**：报告预测，未来云端计算资源的分配模式可能从“固定服务”转向“动态容器”，即由无数个瞬间创建并销毁的Agent容器组成的“蜂群”。
- **行动建议**：报告建议开发者停止编写针对特定模型的修复补丁，转而构建基于沙盒环境的解耦架构，并关注高性能沙盒技术及云端上下文存储方案。
