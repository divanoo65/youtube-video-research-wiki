# Anthropic

**Anthropic** 是一家专注于AI安全与基础模型研发的美国公司，以其提出的 **Managed Agents（托管Agent）架构** 在[[多Agent协作]]领域产生了深远影响。该架构通过将“大脑”（模型规划能力）与“双手”（代码执行沙盒环境）彻底解耦，解决了传统Agent在长程任务中容易崩溃、记忆污染及环境配置冲突等核心痛点，被视为从“个人工具”向“企业基础设施”演进的关键技术路径。

## 核心贡献：Managed Agents 架构

Anthropic 提出的 Managed Agents 架构，核心思想是**将Agent视为无状态的“牛马”而非需要精心呵护的“宠物”**。该架构通过四个关键层级的解耦，实现了高可靠性与高性能的并发任务处理：

### 1. 脑手解耦与无状态化
- **解耦逻辑**：将模型决策（大脑）与代码执行（双手）分离。
- **实现方式**：每一项执行任务都在独立的沙盒环境（如Docker或高性能沙盒）中运行。如果任务出错或环境被污染，系统直接销毁该沙盒并重新拉起一个干净的环境，确保系统稳健性。这从根本上解决了单一Agent因环境配置错误或记忆冗余而崩溃的问题。

### 2. Session 与 Agent 的实体分离
在该架构中，Agent与Session被定义为两个完全不同的实体：
- **Agent（静态模板）**：定义Agent的身份、使用的工具、模型及系统提示词（System Prompt）。它不存储动态数据，仅作为配置模板。
- **Session（动态执行）**：承载对话线程、记忆挂载和文件操作。同一个Agent模板可以同时启动多个独立的Session，互不干扰。这种分离使得“Agent负责我是谁，Session负责我在干什么”成为可能。

### 3. 编排者（Coordinator）模式
通过SDK中的 `coordinator` 字段，Anthropic 揭示了一种“不干活的指挥官”角色：
- **职责**：拆解复杂任务、派发指令给下属Agent。
- **规模**：一个编排者名下可管理多达20个拥有独立Prompt和MCP（Model Context Protocol）服务器的Agent。
- **解决痛点**：解决了单一Agent在长程任务中上下文不足、不断压缩记忆导致“变笨”的问题。编排者通过调度不同Agent分工协作（如：编码、审核、测试），保持每个节点的上下文清晰。

### 4. 上下文解耦（Session Store）
这是架构中最具突破性的部分。通过将Context（上下文）从本地搬到云端（如Redis缓存），实现了Agent的彻底无状态化。
- **功能**：Session可以作为“记忆树”进行Fork（分支）、回滚、克隆或恢复。
- **意义**：编排者可以时刻掌握Agent最聪明的状态，并在需要时回滚或复制该状态以应对特定任务。这被业界视为“Agent的操作系统”。

## 技术影响与实践

- **多模型组合下的Token效率**：通过任务图（Task Graph）预定义规则，实现大小模型的异构组合。多个节点并发执行搜索、编码或写作任务，并根据任务阶段的复杂程度灵活调用不同性能的模型，从而实现最佳的Token效率。
- **计算资源变革**：随着Agent架构向“毫秒级启动沙盒”演进，云端计算资源的分配模式从固定服务转向动态容器，形成由无数个根据需求瞬间创建并销毁的Agent容器组成的“蜂群”。

## 关键语录

> “你要扩大Agent的规模，就要解耦‘脑’和‘手’。” —— 强调模型的进化速度远超代码重构速度，必须通过解耦来应对模型升级带来的代码失效。

> “要把Agent当牛马去用，而不是当宠物去养。” —— 强调在服务器架构中，不应过度依赖单一Agent的环境稳定性，而应利用无状态沙盒实现故障隔离和快速重置。
