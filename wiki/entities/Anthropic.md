# Anthropic

**Anthropic** 是一家专注于人工智能安全与基础模型研发的美国公司。其在多Agent协作领域提出了具有革命性意义的 **Managed Agents（托管Agent）架构**，该架构通过解耦决策层与执行层，解决了长程任务中Agent容易崩溃、记忆污染及环境配置冲突等核心痛点，被视为从“个人工具”向“企业基础设施”演进的关键技术范式。

## 核心贡献：Managed Agents 架构

Anthropic提出的Managed Agents架构，核心思想是实现“脑”（模型规划能力）与“手”（执行代码的沙盒环境）的彻底解耦。该架构包含四个关键层级：

### 1. 脑手解耦与无状态化
- **核心逻辑**：将模型决策（大脑）与代码执行（双手）分离。每一项执行任务都在独立的沙盒（如Docker或高性能沙盒）中运行。
- **无状态化**：如果任务出错或环境被污染，系统直接销毁该沙盒并重新拉起干净的环境。这种设计将Agent从需要精心呵护的“宠物”转变为可随时销毁并重建的“牛马”，极大提升了系统的稳健性。

### 2. Session 与 Agent 的实体分离
在该架构中，Agent与Session被定义为两个完全不同的实体：
- **Agent（静态模板）**：定义Agent的身份、使用的工具、模型及系统提示词（System Prompt）。它不存储动态数据，仅作为配置模板。
- **Session（动态执行）**：承载对话线程、记忆挂载和文件操作。同一个Agent模板可以同时启动多个独立的Session，互不干扰。

### 3. 编排者（Coordinator）模式
Anthropic的SDK中引入了 `coordinator` 字段，定义了一种“不干活的指挥官”角色：
- **职责**：拆解复杂任务、派发指令给下属Agent。
- **规模**：一个编排者名下可管理多达20个拥有独立Prompt和MCP（Model Context Protocol）服务器的Agent。
- **解决痛点**：解决了单一Agent在长程任务中上下文不足、不断压缩记忆导致“变笨”的问题。编排者通过调度不同Agent分工协作（如：编码、审核、测试），保持每个节点的上下文清晰。

### 4. 上下文解耦（Session Store）
这是架构中最具突破性的部分。通过将Context（上下文）从本地搬到云端（如Redis缓存），实现了Agent的彻底无状态化。
- **功能**：Session可以作为“记忆树”进行Fork（分支）、回滚、克隆或恢复。
- **意义**：编排者可以时刻掌握Agent最聪明的状态，并在需要时回滚或复制该状态以应对特定任务。这套架构被业界形容为“Agent的操作系统”。

## 技术影响与实践

Anthropic的架构理念推动了多Agent协作模式的范式转移：
- **多模型组合**：通过任务图（Task Graph）预定义规则，实现大小模型的异构组合。多个节点可并发执行搜索、编码或写作任务，根据任务复杂程度灵活调用不同性能的模型，从而实现最佳的Token效率。
- **计算资源变革**：随着Agent架构向“毫秒级启动沙盒”演进，云端计算资源的分配模式从固定服务转向动态容器，形成由无数个根据需求瞬间创建并销毁的Agent容器组成的“蜂群”。

## 重要语录

> “你要扩大Agent的规模，就要解耦‘脑’和‘手’。” —— 摘自Anthropic 4月8日的博客，强调模型的进化速度远超代码重构速度，必须通过解耦来应对模型升级带来的代码失效。

> “要把Agent当牛马去用，而不是当宠物去养。” —— 强调在服务器架构中，不应过度依赖单一Agent的环境稳定性，而应利用无状态沙盒实现故障隔离和快速重置。

> “Session负责我在干什么，Agent负责我是谁。” —— 形象地解释了静态配置（Agent）与动态运行状态（Session）之间的解耦关系。
