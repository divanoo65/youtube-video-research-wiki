# Anthropic Managed Agents 架构与多 Agent 协作演进深度简报

## 执行摘要

本报告基于对 Anthropic 发布的 Managed Agents 架构及其技术实现的深入分析，探讨了人工智能 Agent 领域从“单体工具”向“企业级基础设施”和“蜂群协作”转型的核心逻辑。研究指出，当前 Agent 发展的标准答案在于**“脑手解耦”**与**“无状态化”**。通过将模型的规划决策能力（大脑）与执行代码的沙盒环境（双手）彻底分离，并引入 Session 与记忆存储的独立管理，开发者能够构建出具备高扩展性、高鲁棒性且能应对长程任务的 Agent 操作系统。这种从“养宠物”到“用牛马”的范式转变，预示着未来计算资源分配将向动态、高并发的 Agent 容器化协作方向变革。

## 核心要点

1. **脑手解耦是规模化 Agent 的前提**  
   传统 Agent 开发中，开发者为了掩盖模型记忆缺陷而编写的复杂补丁代码，在模型升级后往往成为累赘。[[Anthropic]] 提出的解决方案是将模型的大脑（规划决策）与双手（代码执行环境）彻底分离。大脑仅负责发出指令，底层的沙盒环境是无状态、可随时销毁重建的独立单元。这种解耦使得一个 Agent 模板可以派生出无数个相互隔离的执行实例，避免了单体架构中因配置错误或记忆污染导致的全系统崩溃。

2. **从“宠物”到“牛马”的运维范式转变**  
   在 [[Managed Agents 架构]] 中，Agent 不再被当作需要精心照料的“宠物”（与本地环境深度绑定，一旦出错难以恢复），而是被视为可消耗的“牛马”。如果某个沙盒代码跑崩或资源耗尽，协调者会直接销毁该实例并拉起一个干净的沙盒。这种模式消除了单点故障风险，使得数千个 Agent 可以像云原生容器一样被动态调度和弹性伸缩。

3. **四层解耦结构实现完整的状态分离**  
   通过逆向分析 Anthropic SDK 的 `beta` 模块，可以抽象出四层递进结构：第一层 **Agent 模板** 静态定义模型、提示词和工具集；第二层 **Session 实例** 作为动态运行实体，记录每一次对话线程；第三层 **记忆存储（Memory Store）** 不再附着于 Agent 身上，而是作为独立组件可挂载到 Session 上，支持 Fork、回滚和克隆；第四层 **会话存储（Session Store）** 将上下文迁移到云端（如 Redis），使得 Agent 可以在任意节点恢复执行，实现真正的[[无状态化]]。

4. **协调者（Coordinator）驱动多 Agent 蜂群协作**  
   在复杂长程任务中，单一的[[协调者模式]] Agent 不直接参与执行，而是负责将任务拆解为多个子任务并派发给最多 20 个独立的子 Agent（每个子 Agent 拥有独立的提示词和工具）。这种架构解决了传统单一 Agent 因上下文压缩而丢失记忆的问题，并且可以根据任务特点混合使用大模型（如 [[Claude Code]]）和小模型（如 [[Codex]]），实现 Token 效率最优化。例如在编码任务中，可以先用小模型生成草稿，再用大模型进行审阅和优化。

以上核心要点揭示了 [[Anthropic Managed Agents 架构与多 Agent 协作演进深度简报]] 中描述的技术变革方向，为构建下一代企业级 Agent 基础设施提供了清晰的理论支撑和工程实践指南。
