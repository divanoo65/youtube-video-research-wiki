# LLM-Wiki

## 概念定义

LLM-Wiki（又称“小Wiki”）是由人工智能专家安德烈·卡帕西（Andrej Karpathy）提出的一种颠覆性的个人知识库构建范式。它旨在解决传统知识管理中“收藏即吃灰”和信息孤岛的问题，其核心思想是将大语言模型（LLM）的角色从传统的“解释器”转变为“编译器”。在传统的检索增强生成（RAG）模式中，大模型被视作一个临时的解释器：每次查询时，模型从静态的知识库中检索相关片段，然后进行一次性解读，整个过程是线性的、无积累的，导致第一篇笔记无法让第二篇笔记变得更聪明。而LLM-Wiki则要求大模型扮演编译器的角色：它将用户摄入的原始、零散信息（如网页剪藏、读书笔记、会议记录等）进行深度加工和结构化组织，自动生成具备双向链接和统一架构（Schema）的高质量Wiki页面。这些页面不是一次性消费品，而是可被反复引用、更新和交叉连接的知识节点。通过这种编译过程，知识库不再是一个静态的仓库，而是一个动态演化的系统，随着信息量的增加，知识之间的连接越来越丰富，系统本身会变得越来越智能，从而实现知识的“复利增长”。这一理念从根本上改变了个人知识管理的底层逻辑，强调长期价值而非短期便利。

## 技术细节

LLM-Wiki的实现依赖于一套完整的三层架构：

- **原始数据层（Raw Data Layer）**：包含所有未经处理的输入，如通过Obsidian Web Clipper抓取的网页、从DeepSeek或Claude等模型对话中导出的文本、手写笔记扫描件等。这一层只做简单的格式转换和安全过滤，不对内容进行语义处理。
- **Wiki产物层（Wiki Output Layer）**：这是系统的核心成果，由LLM编译器生成的Markdown页面组成。每个页面都遵循统一的[[schema]]，包括Frontmatter（元数据）、结构化正文（定义、细节、应用、相关链接等），并自动包含[[bi-directional-link]]。通过[[dataview]]插件，系统可以基于元数据进行动态查询，生成聚合视图。
- **规则层（Rules Layer）**：指导LLM如何“编译”的规则集，包括Schema规范、链接策略（哪些主题需要自动创建页面）、健康检查脚本（例如检查缺失的链接或孤立的页面）。规则层确保输出的一致性，并支持迭代优化。

工作流通常包括：**摄入** → **查询与回填** → **健康检查** → **人工审核**。例如，用户通过[[obsidian]]插件摄入一篇博客文章后，LLM会自动解析内容，生成若干新页面，并自动链接到已有页面。随后，用户可以通过查询发现缺失的上下文，LLM会调用其他信息源进行回填。定期运行的健康检查（如[[graph-view]]中的孤立节点检测）帮助维护知识库的质量。

## 应用场景

1. **个人学习与研究**：研究者可以将大量论文、笔记投入LLM-Wiki，系统自动生成概念卡、人物卡（如[[andrej-karpathy]]）、方法卡等，并通过双向链接串联各学科知识，形成跨领域的洞见网络。
2. **企业知识库建设**：团队成员将内部文档、会议纪要、项目记录通过LLM-Wiki编译后，新成员可以快速通过结构化页面和链接关系了解组织知识，大幅降低信息传递成本。
3. **写作与内容创作**：作家将灵感碎片、素材库导入系统，LLM-Wiki能自动推荐相关主题并生成草案框架，创作过程变得更具系统性和创新性。
