# 独立评估者模式

**定义**
独立评估者模式（Independent Evaluator Pattern）是 [[Anthropic]] 提出的一种 Agent 系统设计模式，将任务执行拆分为三个独立角色：Planner（规划者）负责拆解任务目标和制定执行计划，Generator（执行者）负责按照计划执行具体操作，Evaluator（评估者）负责独立验收执行结果。核心原则是评估者必须是独立的（不能是同一个 Agent 实例），且必须在真实环境中验证结果（如实际运行页面或代码），而非仅审查生成内容。

**在本库的具体例子**
- 在 [[Harness-Engineering-深度简报-构建稳定-可落地的-AI-Agent系统]] 中，独立评估者模式被列为 [[Anthropic]] 的一线实践，是 Harness 架构中 [[评估与观测层]] 的具体实现方案。视频指出评估者必须是独立的，并且在真实环境中操作——例如评估代码时需要在真实环境中运行，而非仅仅审查代码文本。
- 该模式与 [[执行编排层]] 紧密配合：Planner → Generator → Evaluator 形成完整的执行闭环。当 Evaluator 发现问题时，反馈给 Generator 修正，或在无法修正时回退到 Planner 重新规划路径。

**技术实现细节**
1. **角色隔离**：Planner、Generator、Evaluator 使用独立的模型实例或完全独立的 Agent 进程运行，确保评估结果不被生成过程污染。
2. **真实环境验证**：Evaluator 在隔离的沙箱环境（如 Docker 容器、临时 VM）中执行生成的代码或操作，输出实际验证结果而非仅做静态分析。
3. **反馈循环**：Evaluator 将验证结果（通过/失败及具体问题）结构化反馈给 Generator，Generator 修正后再次提交验证，形成闭环。
4. **Planner 仲裁**：当 Generator 多次修正仍无法通过 Evaluator 验证时，Planner 介入重新制定执行方案或切换路径。
5. **评分标准**：Evaluator 使用可量化的通过条件（如测试用例通过率、代码风格规范检查结果），而非主观判断。

**与近似概念的边界**
- **区别于 [[评估与观测层]]**：独立评估者模式是评估与观测层的一种具体实践方案。后者是 Harness 架构层级的抽象定义，涵盖所有评估和观测机制（包括日志、监控、测试等），而独立评估者模式特指「角色拆分+真实环境验证」这一特定实现。
- **区别于代码审查（Code Review）**：传统代码审查是人工进行的，且审查者仅查看代码文本；独立评估者模式中 Evaluator 是 AI Agent，且必须在真实运行环境中验证代码行为。
