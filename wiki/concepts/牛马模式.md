# 牛马模式

**牛马模式**（Beast of Burden Mode）是 Anthropic 在 Managed Agents 架构中提出的智能体设计范式，它与传统的“宠物模式”相对立，核心思想是将 Agent 视为**无状态的、独立的沙盒执行单元**，每次运行后即可销毁，下次任务时重新从干净环境拉起。这种模式彻底改变了以往将 Agent 记忆、状态与本地环境深度绑定的做法，解决了模型进化快于手动修复代码、记忆堆积导致系统脆弱等根本性问题。

在牛马模式下，Agent 不再是一个需要长期呵护、修补的“宠物”，而是一头可以随时更换的“牛马”。如果某个沙盒出错，系统不会尝试修复它，而是直接销毁并重启一个全新的环境。这种设计带来了两个关键好处：一是**极高的鲁棒性**——错误不会累积，系统始终在处理干净的状态；二是**极佳的可扩展性**——无状态化使得任务可以轻松并行化，每个 Agent 实例独立运行，不受之前执行的影响。同时，由于不需要在 Agent 内部保留长链路记忆，Token 消耗显著降低，推理效率大幅提升。

## 技术细节

牛马模式的实现依赖于 **脑手解耦** 架构，即将大模型的规划决策（大脑）与执行代码的沙盒环境（双手）彻底分离。具体技术细节包括：

1. **无状态化（Stateless）**：Agent 不维护任何本地状态，所有持久化数据（如对话历史、中间结果）都存储在外部系统（如 [[Redis]] 或数据库）中。每次任务开始，协调者创建一个全新的沙盒，从零初始化。
2. **沙盒环境隔离**：每个 Agent 运行在独立的 [[Docker]] 容器或轻量级沙箱（如 [[腾讯Cube Sandbox]]）中，文件系统、环境变量、网络配置均互不干扰。任务完成后容器被销毁，资源释放。
3. **任务图驱动**：协调者根据 [[任务图]] 的节点定义，动态分配 Agent 实例。每个节点对应一个原子任务，Agent 只负责当前任务，不记忆前后上下文。
4. **状态回滚与复制**：当 Agent 失败时，协调者根据 [[状态回滚与复制]] 机制，从之前已保存的检查点重新拉起一个干净 Agent，而不是尝试修复崩溃的 Agent。
5. **混合模型策略**：协调者可根据任务复杂度选择不同的底层模型（如 [[Claude]] 或开源模型），Agent 本身不关心模型选择，只负责执行。

## 应用场景

- **大规模智能体编排**：当需要同时运行数百个 Agent 执行不同子任务（如数据采集、代码审查、客户服务）时，牛马模式确保单个 Agent 的失败不会影响全局，且能快速横向扩展。
- **高频迭代的开发环境**：在 CI/CD 管道中，每个代码提交都可能触发 Agent 进行测试、构建、部署。牛马模式让每个构建在完全干净的沙盒中执行，避免依赖污染。
- **企业级 B2B SaaS 平台**：例如 [[Palantir AIP]] 或 [[OpenAI Frontier]] 的多租户场景，每个用户的 Agent 实例相互隔离，销毁重建不影响其他租户。
- **嵌入式 Agent 系统**：在物联网、边缘计算中，Agent 资源受限，牛马模式允许按需创建、用完即弃，节省存储和计算资源。

## 相关概念

- [[宠物模式]]：与牛马模式相对，强调长期维护、状态绑定，易造成脆弱性。
- [[Managed Agents]]：Anthropic 的官方管理 Agent 架构，牛马模式是其主要设计原则。
- [[脑手解耦]]：实现牛马模式的基础技术，将规划与执行分离。
- [[无状态化]]：牛马模式的核心特性之一。
