# Managed Agents（托管智能体架构）

Managed Agents 是 Anthropic 提出的一种多 Agent 协作架构，其核心理念是将决策层（大脑）与执行层（双手）彻底解耦，使 Agent 从“宠物”模式转变为“牛马”模式，从而解决长程任务中 Agent 容易崩溃、记忆污染及环境配置冲突等问题。该架构被视为“蜂群思维”的版本答案，预示着 Agent 从个人工具向企业基础设施的演进。

## 四层解耦架构

Managed Agents 通过以下四个层级的解耦，实现了高性能、高可靠性的多 Agent 协作：

### 1. 脑手解耦：无状态沙盒执行
传统 Agent 将模型逻辑、上下文记忆与本地环境深度绑定，一旦环境配置出错或记忆冗余便会崩溃。Managed Agents 将大脑（模型决策）与双手（代码执行）分离：每一项执行任务都在独立的沙盒（如 Docker 或高性能沙盒 Cube Sandbox）中运行。如果任务出错或环境污染，系统直接销毁该沙盒并重新拉起干净的环境。这种 **[[无状态化]]** 设计确保了系统的稳健性，让 Agent 可以像“牛马”一样随时被替换而非反复修复。

### 2. Session 与 Agent 实体分离
在该架构中，Agent 与 Session 被定义为两个完全不同的实体：
- **Agent（静态模板）：** 定义身份、工具、模型及系统提示词，不存储动态数据，仅作为配置模板。
- **Session（动态执行）：** 承载对话线程、记忆挂载和文件操作。同一个 Agent 模板可以同时启动多个独立的 Session，互不干扰。这种设计使得“Session 负责我在干什么，Agent 负责我是谁”成为可能。

### 3. 编排者模式（Coordinator）
在 Anthropic SDK 中发现的 `coordinator` 字段，揭示了一种“不干活的指挥官”角色。编排者负责拆解复杂任务、派发指令给下属 Agent，一个编排者名下可管理多达 20 个拥有独立 Prompt 和 MCP 服务器的 Agent。该模式解决了单一 Agent 在长程任务中上下文不足、不断压缩记忆导致“变笨”的问题。编排者通过调度不同 Agent 分工协作（如编码、审核、测试），保持每个节点的上下文清晰，这种多 Agent 协作方式正是 **[[蜂群架构]]** 的具体实现。

### 4. 上下文解耦（Session Store）
这是架构中最具突破性的部分。通过将上下文从本地搬到云端（如 Redis 缓存），实现了 Agent 的彻底无状态化。Session 可以作为“记忆树”进行分支、回滚、克隆或恢复。编排者可以时刻掌握 Agent 最聪明的状态，并在需要时回滚或复制该状态以应对特定任务。这一云端化的 **[[Session Store]]** 机制，使得整个架构可以被视为“Agent 的操作系统”。

## 多模型组合与计算资源变革

在实践中，通过任务图（Task Graph）预定义规则可以实现大小模型的异构组合：多个节点并发执行搜索、编码或写作任务，并根据任务阶段的复杂程度灵活调用不同性能的模型，从而优化 Token 效率。此外，随着架构向毫秒级启动沙盒演进，云端计算资源的分配模式将从固定服务转向动态容器——无数个根据需求瞬间创建并销毁的 Agent 容器组成“蜂群”。

## 核心价值

Managed Agents 架构彻底改变了 Agent 的开发与运维方式。开发者无需再编写针对特定模型的复杂补丁，转而构建基于沙盒环境的解耦架构；企业可将 Agent 视为基础设施而非个人助手，探索处理长程、复杂业务流程的新路径。正如报告中所指出的：要扩大 Agent 规模，就必须解耦大脑和双手，将 Agent 当牛马去用，而不是当宠物去养。
