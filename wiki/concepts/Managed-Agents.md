# Managed-Agents

**Managed-Agents** 是 [[Anthropic]] 提出的一种革命性多Agent协作架构，核心思想是通过将决策层（大脑）与执行层（双手）彻底解耦，实现Agent的**无状态化**和**规模化部署**。该架构将Agent从需要精心维护的“宠物”转变为可随时销毁重建的“牛马”，为复杂长程任务提供了高可靠性的解决方案。

## 核心架构

Managed-Agents 架构由四个关键层级构成，每一层都实现了特定的解耦：

### 1. 脑手解耦
传统Agent将模型逻辑、上下文记忆与本地环境深度绑定，导致环境配置出错或记忆冗余时系统崩溃。Managed-Agents 将大脑（模型决策）与双手（代码执行沙盒）分离。每一项执行任务都在独立的沙盒（如Docker或高性能沙盒Cube Sandbox）中运行。如果任务出错或环境污染，系统直接销毁该沙盒并重新拉起干净的环境，确保系统稳健性。

### 2. Session 与 Agent 实体分离
在该架构中，Agent与Session被定义为两个完全不同的实体：
- **Agent（静态模板）**：定义Agent的身份、使用的工具、模型及系统提示词（System Prompt）。它不存储动态数据，仅作为配置模板。
- **Session（动态执行）**：承载对话线程、记忆挂载和文件操作。同一个Agent模板可以同时启动多个独立的Session，互不干扰。

### 3. 编排者（Coordinator）模式
通过SDK中的`coordinator`字段，揭示了一种“不干活的指挥官”角色：
- **职责**：拆解复杂任务、派发指令给下属Agent。
- **规模**：一个编排者名下可管理多达20个拥有独立Prompt和[[MCP]]（Model Context Protocol）服务器的Agent。
- **解决痛点**：解决了单一Agent在长程任务中上下文不足、不断压缩记忆导致“变笨”的问题。编排者通过调度不同Agent分工协作（如：编码、审核、测试），保持每个节点的上下文清晰。

### 4. 上下文解耦（Session Store）
这是架构中最具突破性的部分。通过将Context（上下文）从本地搬到云端（如Redis缓存），实现了Agent的彻底无状态化。
- **功能**：Session可以作为“记忆树”进行Fork（分支）、回滚、克隆或恢复。
- **意义**：编排者可以时刻掌握Agent最聪明的状态，并在需要时回滚或复制该状态以应对特定任务。

## 关键洞察

### 多模型组合下的Token效率
通过任务图（Task Graph）预定义规则，可以实现大小模型的异构组合：
- **异步协作**：多个节点并发执行搜索、编码或写作任务。
- **成本优化**：根据任务阶段的复杂程度，灵活调用不同性能的模型，从而实现最佳的Token效率（Token Efficiency）。

### 计算资源的变革
随着Agent架构向“毫秒级启动沙盒”演进，云端计算资源的分配模式可能发生变化：从固定服务到动态容器，未来可能不再是长期运行的固定容器，而是由无数个根据需求瞬间创建并销毁的Agent容器组成的“蜂群”。

## 实践启示

1. **架构转向**：开发者应停止编写复杂的、针对特定模型的修复补丁（Harness），转而构建基于沙盒环境的解耦架构。
2. **关注基础设施**：密切关注高性能沙盒技术（如腾讯Cube Sandbox）以及云端上下文存储方案，为大规模Agent并发做好技术准备。
3. **从工具到系统**：企业应将Agent视为基础设施而非简单的个人助手，探索如何在企业内部署Managed Agents架构以处理长程、复杂的业务流程。
4. **优化模型配比**：放弃“全场景单模型”思路，利用编排器实现大小模型组合，通过多节点并发提升任务完成速度与Token使用效率。
