# 查询与回填

## 概念定义

查询与回填（Query & Refinement）是 [[llm-wiki]] 运行三步骤中的核心环节，负责知识库的实时检索与动态增长。它包含两个紧密耦合的动作：**查询**与**回填**。

在查询阶段，系统不依赖复杂的向量数据库或高成本的嵌入计算，而是利用预先生成的纯文本索引文件 `Index.md` 实现毫秒级的全域内容定位。这个索引由 [[ingestion]] 步骤动态维护，以简单的 Markdown 列表罗列所有概念页面及其关键词、摘要和双向链接，支持通过字符串匹配或正则表达式进行快速过滤。由于所有内容均为纯文本且结构化，检索结果天然可读、可解释，用户或 AI 能够直观地知道从哪个页面获得了上下文。

在回填阶段，系统将每一次与用户的对话结果（尤其是高质量的回答、推理链条或修正信息）自动提取并写回到 Wiki 文件夹中。具体而言，AI 会将对话中产生的结构化知识（如新概念、现有页面的补充说明、矛盾澄清、数据更新等）转化为符合 [[schema]] 的 Markdown 文件，存入原始文件夹或直接追加到已有页面。这样一来，每次交互不再是一次性的消耗性过程，而是转化为知识库资产的长期积累——使用越频繁，知识库越丰富，从而触发 [[knowledge-compounding]] 效应。

查询与回填共同保证了 LLM-Wiki 作为“编译器模式”知识库的两个关键特性：**低延迟检索**（毫秒级定位）和**资产增值**（对话结果自动沉淀）。它与传统 RAG 的“解释器模式”形成鲜明对比：后者每次检索都是瞬时的、孤立的，无法积累历史优化；而前者通过回填机制将每一次交互变成对知识库的“编译”，使得知识随使用不断增值。

## 技术细节

- **检索机制**：核心依赖 `Index.md` 文件，该文件由摄取步骤定期更新，结构为 Markdown 列表，每项包含页面名、摘要、标签、最近修改时间等元数据。检索时采用字符串匹配或自定义脚本（如基于 `grep` 或 `awk`），无须启动任何服务或数据库。
- **回填流程**：AI 在回答后，系统会解析对话中的内容变化：若发现新概念，则创建新文件（自动填充 frontmatter 和正文）；若为已有概念补充，则更新对应文件的正文或 frontmatter；若检测到矛盾或过时信息，则添加 `conflict` 或 `deprecated` 标签并记录来源。所有修改均通过版本控制（如 Git）跟踪，支持人工复核。
- **性能保障**：由于无需实时向量化，查询时延主要来自文件 I/O，通常在 1~5 毫秒。百万级页面规模下，通过分片索引或索引缓存仍可保持亚秒级响应。
- **一致性维护**：回填操作会触发健康检查脚本，对修改后的页面进行格式校验、链接完整性检查和冲突标记，确保知识库整体有序。

## 应用场景

1. **个人知识管理**：用户日常记笔记、写作时，AI 助手通过查询快速定位相关页面，提供上下文建议；每次修改或补充的内容自动回填，笔记库每日自动成长，避免知识碎片化。
2. **团队协作 Wiki**：多人同时编辑同一知识库，AI 在回答共性问题时自动将答案回填到适当的概念页面，减少重复劳动，并保留讨论的上下文和决策依据。
3. **持续学习系统**：用户阅读新资料（如论文、书籍）后，AI 将摘要和与已有知识的关联自动回填，形成动态更新的学习笔记——第一篇笔记不会让第二篇变聪明，但回填机制让第十篇笔记直接继承前九篇的洞察。
4. **AI 对话增强**：任何与 LLM 的对话都可以通过回填转变为知识库的“微迭代”，例如纠错、补充案例、添加溯源链接等，使对话本身成为知识生产的源头。
