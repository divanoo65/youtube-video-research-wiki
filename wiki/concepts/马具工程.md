# 马具工程（Harness Engineering）

## 定义
马具工程（Harness Engineering）是 AI 工程化的第三阶段，核心定义是 **Agent = Model + Harness**。Harness（马具/约束装置）指模型之外所有决定其稳定交付的机制，包括信息边界、工具系统、执行编排、记忆状态、评估观测、约束校验与恢复。其本质是**系统设计**——用确定性的工程结构包裹概率性的模型，将任务成功率从不足 70% 提升到 95% 以上。

## 在本库的具体例子

- [[Harness-Engineering-从概率生成走向稳定交付]] 视频是马具工程的核心来源，详细阐述了六层架构：
  - **信息边界层**：角色目标定义、信息裁剪与结构化组织。
  - **工具系统层**：工具选择、调用时机控制、结果提炼反馈。
  - **执行编排层**：任务步骤串联，构建“理解-判断-补充-生成-检查-修正”轨道。
  - **记忆与状态管理层**：当前任务状态、中间结果、长期记忆与用户偏好。
  - **评估与观测层**：独立验收机制、日志与指标归因。
  - **约束、校验与恢复**：操作红线、失败重试、路径回滚。
- [[Anthropic]] 的 Context Refresh 和角色分离机制是马具工程在顶级公司的落地。
- [[OpenAI]] 的环境设计者、模块化指令集和自动治理系统也是马具工程的典型体现。

## 技术实现细节

1. **评估层独立于生成层**：不使用模型自评，而是创建独立的 Evaluator 实例或外部校验工具（如代码运行、UI 截图对比）。
2. **状态恢复点**：在长任务中定期存储中间状态，失败时可回滚到最近稳定点而非从头开始。
3. **准入准出约束**：在每个步骤之前检查输入（准入），之后验证输出（准出），不符合条件则触发重试或切换路径。
4. **工具结果提炼**：对 API 返回的冗余信息进行结构化提炼后再传给模型，避免信息污染。

## 与近似概念的边界

- **与提示工程**：提示工程仅关注提示措辞，马具工程涵盖整个运行系统。提示工程是马具工程中信息边界层的子集。
- **与上下文工程**：上下文工程关注信息供给，马具工程在此基础上增加了工具、编排、记忆、观测、约束等维度。
- **与传统软件工程**：传统软件工程中的监控、日志、容错等概念被继承，但马具工程特别针对语言模型的不确定性和自评偏差设计了独立验证机制。
