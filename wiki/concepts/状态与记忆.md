# 状态与记忆

## 概念定义

状态与记忆是指 Agent 系统在执行长链路、多阶段任务时，通过显式或隐式的方式保存当前进度、上下文信息以及用户的个性化偏好，从而确保任务中断后能够从断点处继续执行，并在后续阶段复用已经积累的知识。在 [[Harness 实践：Agent 全自动知识讲解视频生成技术简报]] 所描述的视频生产流水线中，状态与记忆的核心载体是一个名为 `outline.md` 的开发计划文件。该文件不仅记录了视频内容的章节结构、视觉步骤的标注，还动态记录了用户在审核环节中提出的修改偏好（例如口播风格调整、某一段落的节奏要求等）。通过这种方式，Agent 能够在四个生产阶段（内容编写与规划、工具执行、人工审核、最终合成）之间保持连贯的语义上下文，避免因为阶段切换或工具调用的中断而丢失关键信息。状态与记忆不是简单的快照，而是结合了结构化文档（`outline.md`）和 Agent 内部工作记忆（token‑level 上下文）的混合机制，使得即使 Agent 被重启或切换到不同模型，仍然能够通过读取 `outline.md` 快速重建当前的状态。

## 技术细节

状态与记忆的实现依赖于两个层次：

1. **显式持久化（文件级别）**  
   Agent 将整个生产流程的当前进度、关键决策和用户反馈写入 `outline.md`。该文档采用 Markdown 格式，包含以下结构化字段：
   - `## 章节进度`：标记已完成/进行中的章节。
   - `## 用户偏好记录`：记录每一次人工审核后确认的风格要求（如“使用第二人称”“避免技术术语”）。
   - `## 依赖状态`：记录已生成的音频文件路径、视频片段索引等资源引用关系。
   - 每次阶段切换或 checkpoint 触发时，Agent 会主动更新该文件；恢复时首先读取 `outline.md` 重建环境。

2. **隐式记忆（上下文窗口）**  
   在单一 Agent 会话内，Agent 通过 LLM 的上下文窗口（如 Claude 的 200K token）保留历史对话及工具调用记录。但为了避免注意力稀释，系统采用 [[上下文管理]] 策略，仅在当前阶段加载相关的子文档（如口播规范、开发指引），而非全量历史。当显式记忆与隐式记忆冲突时，以 `outline.md` 中的记录为准，保证跨 session 的一致性。

3. **偏好归约机制**  
   对于用户的每次修改请求，Agent 不仅执行修改，还会将修改的“意图”抽象为一条偏好规则（例如“所有章节的过渡句都要加 1 秒停顿”），归约后写入 `outline.md` 的 `## 用户偏好记录` 段。后续阶段在生成内容时自动应用这些规则，实现“一次调优，全局生效”。

## 应用场景

- **多阶段视频生产**  
  在 [[Harness]] 框架下，视频生产被拆分为内容编写、视觉设计、语音合成、最终渲染四个阶段，每个阶段之间需要人工审核点。状态与记忆机制允许 Agent 在审核后根据反馈精准调整后续输出，而无需重新从头生成。例如，用户在第一阶段指出“语速太快”，该偏好会被记录并应用于第二阶段的语音合成参数，无需手动修改配置文件。

- **跨会话协作与容错**  
  当 Agent 因超时、模型切换（如使用 [[CC Switch]] 从 MiniMax 2.5 切换到 2.7）或意外中断时，新的 Agent 实例可以通过读取 `outline.md` 快速恢复到上次 checkpoint 的状态，并继续执行“最小切片修复”级别的小范围修改，避免全量重做。这在长链路（如 30 分钟视频）生产中显著降低了时间和 token 成本。

- **个性化定制流水线**  
  不同用户对口播风格、动画偏好、字幕样式等有不同要求。状态与记忆使得 Agent 能够持续学习并记住每个项目的专属规则，甚至在同一个 [[TMUX]] 多终端环境内多个并行项目之间保持隔离的偏好记忆。这对于面向客户的定制化视频生成服务尤其有价值。
