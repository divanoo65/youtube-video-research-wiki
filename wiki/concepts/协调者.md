## 概念定义

协调者（Coordinator）是 Managed Agents 四层解耦架构中位于第三层“编排层”的核心实体，其角色定位为不直接承担具体执行工作的“指挥官”。与第一层静态模板 Agent（定义“我是谁”）和第二层动态会话 Session（定义“我在干什么”）不同，协调者的主要职责是围绕任务全生命周期进行规划、拆解、分配与监控。在长程任务场景中，单一 Agent 常常因上下文窗口限制而被迫压缩记忆，导致关键信息丢失；协调者的引入从根本上改变了协作范式——人类从手动协调者的角色中解放出来，由专门的 Agent 担任这一职能。协调者能够同时管理多达 20 个具有独立工具集和自定义 MCP 服务器的子 Agent，每个子 Agent 都运行在隔离的沙盒环境（如 Docker 容器）中，实现大脑与执行环境的物理解耦。协调者不仅负责生成任务图（如“编码 → 评审 → 测试”的流水线），还能识别并记录子 Agent 在其生命周期中表现出的“最聪明状态”，从而在需要时执行状态回滚或克隆操作，以最优状态应对特定子任务。这种设计使得多 Agent 协作具备高度的弹性和可观测性。

## 技术细节

协调者的实现依赖于几个关键技术机制：

- **任务图驱动调度**：协调者根据用户意图或预定义工作流，生成有向无环图（DAG）形式的任务图。任务图中的每个节点代表一个子任务，边代表执行依赖关系。协调者根据子 Agent 的专长、当前负载及可用工具集动态分配节点，支持并行调度与顺序依赖。
- **状态回滚与复制**：协调者维护每个子 Agent 的执行快照（包括上下文、记忆挂载文件、中间输出）。当子 Agent 在任务执行过程中偏离预期或效率下降时，协调者可选择回滚到之前记录的最优状态，或克隆该状态到另一个空闲 Agent 实例中继续执行，避免从头开始。
- **多 Agent 的独立运行时隔离**：每个子 Agent 拥有独立的工具集合、MCP 服务器（如 Model Context Protocol 定义的模型-工具通信协议）和内存空间。协调者通过统一的 API 网关与子 Agent 通信，确保任务派发与结果收集的低延迟和一致性。
- **会话持久化与云端记忆**：协调者与第四层持久层（Session Store，如 Redis）深度集成。子 Agent 的上下文（Context）实时同步到云端，支持 Fork（分叉）、回滚和克隆操作，使得整个 Agent 体系彻底实现无状态化——协调者可以随时创建、销毁或重启子 Agent 而不丢失关键上下文。

## 应用场景

- **多阶段软件工程流水线**：在代码生成、代码评审、自动化测试、部署验证的长链任务中，协调者分别调度具有编码技能、评审技能和测试技能的子 Agent。每个子 Agent 运行在独立的沙盒环境中，协调者监控各阶段输出，在发现测试失败时可回滚到“编码完成”状态并要求子 Agent 重新修复。
- **复杂决策支持系统**：例如企业级数据分析场景，协调者将用户的问题拆解为数据查询、洞察生成、可视化展示等子任务，指派给不同专长的 Agent（如 SQL 专家 Agent、图表生成 Agent），并确保最终结果的连贯性与一致性。
- **智能体蜂群架构的编排核心**：在类 Palantir AIP 或 OpenAI Frontier 的多智能体协作平台中，协调者充当蜂群的“蜂王”，负责动态招募、解聘和替换 Worker Agent，同时维护全局目标与局部任务的映射关系。
