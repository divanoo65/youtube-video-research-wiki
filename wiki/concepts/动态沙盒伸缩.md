# 动态沙盒伸缩

**动态沙盒伸缩** 是一种在 [[多Agent协作架构]] 中将执行环境容器化，实现按需创建、使用后立即销毁的弹性资源管理策略。其核心思想源自分布式系统中的“宠物 vs 牛马”模式：不再像养宠物一样维护一个长期运行、状态复杂的 Agent 进程，而是将每个子任务分配给一个临时、无状态的沙盒环境，任务完成后直接销毁该沙盒，需要时再从干净模板快速重建。

## 背景与动机

在传统的 Agent 开发中，开发者往往需要编写大量补丁代码（Harness）来弥补模型记忆缺陷或控制执行环境。随着模型版本升级，这些修补代码迅速过时，整个系统变得脆弱易崩。Anthropic 工程师在构建早期 Agent 时曾为此编写数千行 Harness 代码，但随模型迭代这些代码瞬间失效。这揭示了直接“修补”模型执行环境的死胡同：**必须将模型的大脑（规划决策）与双手（执行沙盒）彻底解耦**。

动态沙盒伸缩正是解耦后的执行层管理方案。它允许系统在没有任何持久化状态的前提下，为每个子任务或每个 Agent 工作单元快速分配一个干净的容器，任务或会话结束后立即回收资源。这种模式彻底规避了环境冲突、配置漂移等问题，使大规模 Agent 蜂群协作成为可能。

## 技术实现细节

1. **容器化沙盒**  
   最常见的实现是使用 Docker 容器，但大规模并发场景下 Docker 的秒级启动可能成为瓶颈。报告中引用了部分研究机构采用 **Cube Sandbox**（腾讯云开源）等毫秒级启动技术，配合轻量内核隔离，实现近乎瞬时的沙盒创建。

2. **无状态化**  
   沙盒内部只负责执行接收到的指令（如代码编写、文件操作、网络请求），**绝不保留会话记忆**。记忆与上下文被外部化到 [[Session Store]]（如 Redis）中，沙盒可以随时销毁而不会丢失任务进度。这种设计使得系统可以从任意故障中恢复：只需丢弃当前沙盒，从模板新建一个，再挂载上次保存的上下文即可。

3. **编排器的调度**  
   动态伸缩的触发完全由 [[Coordinator（编排器）]] 控制。它根据任务图（Task Graph）中的节点依赖关系，决定何时创建新的沙盒 Worker、何时需要销毁空闲沙盒。例如，在代码生成任务中，Coordinator 可以同时拉起 20 个沙盒：5 个用于并发搜索，5 个用于编码，5 个用于单元测试，5 个用于代码审查。一旦某个任务完成，对应的沙盒立即销毁，释放计算资源供其他任务使用。

## 在 Managed Agents 架构中的角色

动态沙盒伸缩是四层解耦模型（Agent 沙盒 → Session 层 → Session Store → Coordinator）中**第一层（Agent 沙盒）** 的核心实践。它通过容器化实现了：

- **稳定性**：沙盒崩溃不影响其他沙盒，Coordinator 只需重新创建一个干净的沙盒并恢复会话上下文。
- **弹性**：计算资源随 Agent 实例数量动态分配，峰值时自动扩容，低谷时自动缩容，契合云端按需计费模式。
- **审计与隔离**：每个任务在独立的沙盒中执行，日志和文件系统相互隔离，方便事后溯源和安全性检查。

## 典型例子

报告中描述了这样一个未来场景：用户通过手机或眼镜发出一个复杂指令（例如“根据上周销售数据生成一份包含图表和竞品分析的季度报告”），云端 Coordinator 会瞬间创建数十个沙盒：

- 1 个沙盒用于数据查询
- 1 个沙盒用于调用大模型生成文本
- 1 个沙盒用于渲染图表
- 1 个沙盒用于编译最终 PDF

所有这些沙盒同时工作，任务完成即销毁，整个流程可能在几十秒内结束。用户无需关心底层用了多少计算资源，只需为实际使用的“沙盒秒数”付费。
