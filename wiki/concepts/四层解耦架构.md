# 四层解耦架构

四层解耦架构是 Anthropic 推出的 **Managed Agents**（托管 Agent）体系的核心设计模式，旨在将大模型的“大脑”与执行环境、记忆状态、协作调度彻底分离。该架构将 Agent 从脆弱的“宠物”模式转变为可随时销毁、无状态的“牛马”模式，是实现大规模 Agent 蜂群协作的基础设施级抽象。

## 架构层次详解

### 第一层：[[Agent沙盒 (Sandbox)|Agent沙盒]]
将模型执行环境容器化，实现执行层的彻底独立。传统 Agent 开发中，开发者需要编写大量修补代码（Harness）来弥补模型的记忆缺陷——例如给模型预设工具、维护文件系统状态等。这些修补代码随着模型升级迅速失效。沙盒层通过 Docker 容器或毫秒级启动的轻量沙盒（如 [[Cube Sandbox]]）剥离执行环境：每个 Agent 实例运行在独立、干净的沙盒中，若代码跑崩或配置冲突，直接销毁并重新拉起。参考报告中提到“大脑只管发指令，底层的‘双手’是无状态的独立沙盒”，正是这一层的实践。

### 第二层：[[会话层 (Session)|会话层]]
将 Agent 的**定义**（静态模板）与**执行过程**（动态 Session）分离。一个 Agent 模板（例如“代码审查 Agent”的配置、工具列表、系统提示）可以对应多个独立的 Session 实例。每个 Session 维护自己的执行上下文、中间变量和交互历史，但 Agent 定义本身保持不变。这样开发者可以复用 Agent 定义的同时，为不同任务隔离会话状态，避免相互污染。例如，同一个编码 Agent 可以同时处理两个独立的需求评审 Session，互不干扰。

### 第三层：[[Session Store (上下文管理)|Session Store]]
记忆不再长在 Agent 身上，而是作为外部挂载的存储层。Session Store 支持上下文的 **Fork（分支）**、回滚、克隆和状态恢复。这意味着开发者可以像操作 Git 分支一样操作 Agent 的执行历史：如果某个子任务出错，可以回滚到之前的检查点，或者从某个 Session 分支出一个新的探索路径，而不影响主流程。这种外部化上下文管理直接解决了长程任务中的“上下文丢失”和“记忆混淆”问题。参考报告中的描述：“记忆不再长在 Agent 身上，而是作为外部挂载”，这句话突显了存储层的解耦价值。

### 第四层：[[Coordinator编排器|Coordinator（编排器/协调者）]]
不干活的指挥官。Coordinator 负责拆解用户意图为子任务，派活给最多 20 个 Worker Agent，并在 Worker 之间传递中间结果，管理长程任务中的上下文压缩。参考报告中强调：“编排器/协调者不干活，只负责拆任务、派活、上下文管理”。在实际演示中，Coordinator 会启动多个并发的 Agent（如编码、Review、测试节点），形成一个**任务图**；当某个环节的 Token 消耗过高时，Coordinator 可以动态切换为轻量模型进行并发搜索，强模型负责汇总。这种调度机制避免了 Worker Agent 自行协调时的复杂性和死锁风险。

## 架构的范式转变：从“宠物”到“牛马”

四层解耦架构的核心哲学是“把 Agent 当牛马去用”。在宠物模式下，开发者需要精心维护每一个 Agent 实例的完整状态，像照顾宠物一样担心环境污染、上下文错乱。而在牛马模式下，底层的每个沙盒都是无状态的，Session 和上下文存储在外部存储中，Coordinator 负责群体调度——任何沙盒崩溃都可以像丢弃一头坏掉的牛马一样，直接销毁，重新拉起一个干净的。参考报告中的引言：“哪怕某个沙盒代码跑崩了，配置冲突了，没关系直接干掉，重新拉起一个干净的沙盒接着干”——这正是分布式系统中“宠物 vs 牛马”概念的工程化落地。

## 技术实现路径建议

1. **引入高速沙盒**：Docker 在大规模并发时启动耗时可能成为瓶颈，推荐采用毫秒级启动的沙盒技术（如 Cube Sandbox），以支撑蜂群协作中频繁的销毁与重建。
2. **外部化上下文管理**：将 Session Store 部署在 Redis 或类似高性能缓存中，使 Agent 实现彻底无状态化，支持随时快照恢复、分支和回滚。
3. **构建编排层**：不要将协调逻辑内嵌到 Worker Agent 中，而是开发专门的 Coordinator 服务，通过预定义规则（如任务依赖图、Token 预算）管理多 Agent 并发。参考报告中的演示，Coordinator 可以在不同阶段切换模型大小以优化 Token 效率。

## 行业影响

四层解耦架构正在推动 Agent 从“个人工具”向“企业级基础设施”转型。未来的 Agent 使用场景将变为：用户通过手机或眼镜发出指令，云端瞬间创建数十个独立沙盒，由 Coordinator 调度协作完成任务，任务结束即销毁全部沙盒。这种“随用随建”的模式将彻底重塑计算资源定价模型，同时催生 [[FDE (Forward Deployed Engineer)]] 等新职业，负责将这类复杂系统落地到具体企业场景。
