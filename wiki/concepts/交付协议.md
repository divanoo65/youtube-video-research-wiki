## 概念定义

交付协议（Handoff Protocol）是指在智能体（Agent）工作流中，用于标准化传递当前会话上下文、开发意图、中间产物及决策依据的一套规则与数据格式。它解决了传统 AI 开发模式中“会话孤岛”的问题——当任务从一个智能体流转到另一个（例如从原型设计智能体移交到编码智能体）时，关键信息（如用户偏好、已排除的方案、后续步骤）往往丢失，导致重复劳动或逻辑断裂。交付协议通过将逻辑“实例化”为终端应用，使得 AI 能够提供可运行的、可验证的逻辑雏形。具体而言，`handoff` 的作用非常直接：打包当前会话，写入一个文档，使其他智能体或会话能够很好地使用它。这不仅仅是原始日志的复制，而是经过提炼的知识资产——原生日志（如 JSONL 格式）虽然如实记录了对话情况，但缺乏劳动工具所需的附加功能，如对会话进行总结摘要。交付协议正是根据用户的意图对信息进行了二次加工，使交接内容具备可读性、可追溯性和可执行性，成为智能体间协作的“标准契约”。

## 技术细节

交付协议的核心机制包含三个层面：

1. **上下文打包**：在会话结束时，协议会捕获当前对话中的所有关键状态——包括提出的多方案变体、用户对某个原型的偏好、被放弃的思路及原因、以及下一步的明确意图。打包过程不是原样转储，而是由智能体根据预设模板进行结构化摘要。
2. **格式约定**：协议通常采用 Markdown 作为主要媒介（因为在可读性与机器可解析性之间取得平衡），辅以结构化头部（YAML front-matter）明确移交目标、任务类型、依赖项和已知限制。JSONL 等纯数据格式则作为补充附件，保留完整原始记录用于审计。
3. **意图传递**：协议中必须包含一个“待办”或“任务描述”字段，明确告知接收方智能体应继续执行的动作（例如“将附件中的原型代码用 React + TypeScript 实现”）。该字段由人类在调用 `handoff [参数]` 时提供，也可由原智能体根据对话推理自动填充。

在实践层面，开发者可以通过 `npx` 命令快速集成 `handoff` 技能（如 Matt Pocock 发布的技能包），将其以符号链接形式关联到 [[Claude Code]] 等智能体工具。安装后，工作流可明确分为三个阶段：探索阶段（使用[[原型技能]]生成多方案变体）、执行阶段（基于选定方案编码）、交付阶段（调用 `handoff` 描述后续意图并生成交接文档）。

## 应用场景

- **跨智能体协作**：当设计 Agent 完成 UI 原型后，调用交付协议将设计决策（颜色方案、布局逻辑、用户流）打包移交给编码 Agent，后者可直接读取并生成对应组件的生产代码，避免重复沟通。
- **长时间会话延续**：对于复杂项目，单一会话可能因上下文过长而智力下降。交付协议允许人类在关键里程碑处中断会话，生成一份语义丰富的总结文档，下次开启新会话时由智能体快速恢复语境。
- **团队知识沉淀**：交付协议不只是智能体间使用，也可输出为人类可读的技术文档。例如，将一次多方案发散与收敛的过程记录为“设计决策档案”，供团队成员 review 或作为后续迭代的参考资料。这与 [[Agent Skills 深度解析]] 中提出的“选择而非生成”理念相吻合——协议强调记录哪些方案被否决及其理由，从而使最终交付的成果承载了完整的决策链条。
