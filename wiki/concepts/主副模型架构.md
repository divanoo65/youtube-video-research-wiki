## 概念定义

主副模型架构是 Hermes Agent 在构建复杂 AI 智能体时采用的一种分层模型调度策略。其核心思想是将智能体工作流中的任务划分为“核心推理任务”与“辅助支持任务”两类。主模型（如高性能闭源模型或大型开源模型）负责处理需要深度理解、长上下文推理或创造性输出的核心环节；而副模型（如 Gemini 1.5 Flash、轻量级开源模型）则接管所有标准化、低复杂度、高频次执行的辅助任务，包括但不限于工具调用审批、上下文压缩、视觉摘要提取、MCP 协议中间件处理等。这种架构不仅显著降低了总体的 Token 消耗和 API 调用成本，还通过任务隔离提升了系统的稳定性和响应速度。在实际部署中，用户可以通过配置文件（如 Hermes Agent 的 `config.yaml`）将 `approval`、`compression`、`mcp`、`vision` 等模块的模型 ID 统一指向廉价的副模型，而主模型则保留在 `llm` 字段中。这种设计使得 Hermes Agent 能够在保持强大功能的同时，实现近乎线性的成本控制，特别适合需要大规模并发或长期运行的 Agent 服务场景。

## 技术细节

主副模型架构的实现依赖于 Hermes Agent 的模块化配置系统。在 `config.yaml` 或其他配置文件中，每个辅助模块（如 `approval`、`compression`、`mcp`、`vision`）都可以独立指定模型 ID。例如：

```yaml
llm:
  model: "minimax-m2.7"  # 主模型：高性能模型
approval:
  model: "gemini-1.5-flash"  # 审批模块使用廉价副模型
compression:
  model: "gemini-1.5-flash"  # 上下文压缩使用廉价副模型
mcp:
  model: "gemini-1.5-flash"  # MCP 中间件使用廉价副模型
vision:
  model: "gemini-1.5-flash"  # 视觉理解使用廉价副模型
```

这种配置机制将副模型与主模型完全解耦，允许用户根据任务特性混合使用不同的模型提供商。副模型通常具有更快的推理速度和更低的成本（如 Gemini 1.5 Flash 的定价约为 Gemini 1.5 Pro 的 1/10），且在处理标准化任务时性能足够。此外，主副模型之间通过 Hermes Agent 内置的调度器进行异步通信，主模型生成的中间结果会由副模型进行后处理（如压缩、摘要），从而进一步节省主模型的上下文窗口开销。

## 应用场景

- **企业级 Agent 部署**：在需要同时处理大量用户请求的生产环境中，主副模型架构可以显著降低 API 调用费用。例如，客服 Agent 的主模型负责生成回答，而副模型负责审批工具调用、压缩对话历史、提取用户意图，从而将整体成本降低 60%–80%。
- **长链智能体工作流**：当 Agent 需要执行多步骤任务（如数据爬取、分析、报告生成）时，副模型可以独立处理中间步骤的验证和结果格式化，避免主模型反复被唤醒。
- **视觉与多模态场景**：对于需要处理图片/视频的 Agent，副模型（如 Gemini 1.5 Flash 的视觉能力）可以先行提取关键帧或生成描述，主模型仅需基于结构化文本进行推理，大幅减少视觉 Token 的消耗。
