# 主副模型架构

## 定义
主副模型架构（Primary-Secondary Model Architecture）是一种多模型协作策略：主模型（Primary Model）负责需要深度推理、复杂生成的核心任务，而副模型（Secondary Model）负责一系列简单、辅助性的任务，如任务批准、上下文压缩、记忆刷新、MCP 调用等。通过将辅助任务导向低成本模型，可以显著降低 Token 消耗总量，同时不影响核心任务的质量。

## 技术实现细节
在 Hermes Agent 中，主副模型架构通过配置文件实现。用户需要在配置中指定 `primary_model` 和 `auxiliary_tasks` 部分。`auxiliary_tasks` 包含了多个辅助任务类型，每个任务类型都可以单独指定模型；最佳实践是统一将所有任务指向同一个低成本模型（如 Gemini 1.5 Flash）。配置示例：
```yaml
primary_model: gpt-4o
auxiliary_tasks:
  approval:
    model: gemini-1.5-flash
  compression:
    model: gemini-1.5-flash
  memory_refresh:
    model: gemini-1.5-flash
  mcp_call:
    model: gemini-1.5-flash
  session_search:
    model: gemini-1.5-flash
  vision:
    model: gemini-1.5-flash
  web_tool:
    model: gemini-1.5-flash
```
修改配置文件后需重启 Hermes Gateway 使配置生效。

## 与近似概念的边界
- **模型蒸馏**：模型蒸馏是将大型教师模型的知识压缩到小型学生模型，训练出一个较小的模型直接替代大模型。主副模型架构并非训练压缩，而是运行时将任务分流，保持主模型不变，另设副模型并行协作。
- **任务路由**：类似概念是任务路由系统（如 API Gateway 根据任务类型分发给不同模型），但主副模型架构特指 Agent 内部辅助任务与主任务的静态分流策略，而非动态路由。
- **模型级联**：模型级联是多模型按复杂度依次尝试的架构，但主副模型架构是固定的双模型分工，不存在级联回退路径。
- **混合专家模型（MoE）**：MoE 是模型内部架构（多个 expert 子网络由门控选择），主副模型架构是 Agent 层面的外部模型调度策略，二者不在同一抽象层次。
