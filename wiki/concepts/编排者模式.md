# 编排者模式

编排者模式（Orchestrator Pattern）是一种在 AI Agent 系统设计中广泛采用的架构范式，其核心思想是将**大脑（规划与决策）**与**双手（执行与沙盒）**彻底解耦，通过一个中心化的编排器来协调多个子 Agent、工具或外部服务，共同完成复杂的、长程的任务。在该模式中，编排器负责接收用户意图、将任务拆分为可执行的子步骤、根据步骤类型选择合适的执行单元（如专用 Agent、API 调用、代码沙盒等），并在执行过程中持续收集反馈、处理异常、调整后续计划。这种设计借鉴了分布式系统和微服务架构中的编排理念，但特别针对大语言模型的非确定性、长上下文的记忆管理以及环境容错等挑战进行了优化。

编排者模式并非简单的函数调用链，它要求编排器具备动态规划能力——即根据实时输出重新评估任务分解。例如，当某个子 Agent 返回的结果与预期不符时，编排器可以动态插入新的验证步骤或重试逻辑，而非机械地执行预定义的工作流。为了实现这一点，编排器通常维护一个“记忆树”结构（参见[[记忆树状管理]]），将任务图、中间结果、关键决策点以及环境状态以结构化方式持久化，从而支持断点续跑、分支探索和人工干预。与此相对的另一种模式是[[蜂群模式]]，后者采用去中心化的协调方式（如消息传递、共享黑板），而编排者模式则更适合需要强一致性、严格控制流程以及对异常处理要求较高的场景。

技术细节上，编排者模式通常包含以下组件：
- **编排器实例**：作为单一入口，运行在可信的、有状态的长驻进程中，负责维护任务图（Task Graph）和全局上下文。
- **执行沙盒**：无状态的、轻量级容器或进程（如 [[Docker]] 容器或 [[Cube Sandbox]]），每个子任务在干净的沙盒中独立执行，执行完毕后沙盒即被销毁，避免环境污染。
- **内存与存储层**：通过外部持久化存储（如向量数据库、键值存储）保存会话状态、任务历史以及中间产物，使得编排器可以在故障后重建上下文。
- **通信协议**：编排器与子 Agent 之间通常采用标准化指令格式（如 JSON-RPC、结构化消息），确保双方协议可互换、可审计。

应用场景十分广泛，尤其适合以下领域：
- **复杂文档处理**：如合同审阅，编排器先调用 OCR Agent 提取文本，再调用抽提 Agent 定位条款，接着调用合规 Agent 进行比对，最后生成摘要。
- **自动化代码审查与修复**：编排器将代码仓库扫描任务分发给静态分析 Agent、安全审计 Agent 以及重构建议 Agent，合并结果后提交 Pull Request。
- **多步客户服务工单处理**：一个编排器根据用户问题类型，依次调度意图分类 Agent、知识库检索 Agent、情感分析 Agent 以及答案生成 Agent，并确保最终回答不包含矛盾信息。
- **研究型工作流**：例如生成行业报告，编排器规划子任务：数据爬取、统计分析、图表生成、语言润色，每个子任务由专用 Agent 执行，编排器负责确认每个环节的输出质量并决定是否重跑。

编排者模式与 [[Anthropic Managed Agents 架构深度解析与实践简报]] 中提出的“脑手分离”和“四层解耦”理念高度契合，是实现 Agent 从“宠物模式”向“牛马模式”转型的关键实践之一。通过将决策与执行分离，并采用无状态化的执行单元，编排者模式显著提升了系统的鲁棒性、可扩展性和运维效率。
