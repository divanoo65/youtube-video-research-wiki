## 概念定义

辅助任务是 Hermes Agent 主副模型分离架构中的核心概念之一。在典型的 AI Agent 系统中，不同任务对模型能力的需求差异巨大：复杂逻辑推理、多步决策、深度分析等任务需要高性能（通常成本较高）的模型来保证质量，而大量的辅助性、重复性操作——如任务审批确认、中间结果的压缩与格式化、记忆模块的定期刷新、MCP 工具的简单调用、会话历史检索、技能列表管理、视觉识别中的基础分类、网页内容的结构化提取等——这些任务本身技术难度低、对模型创造性要求不高，但如果全部交由主模型处理，会造成严重的 Token 浪费和响应延迟。辅助任务正是为了将这类“脏活累活”剥离出来，统一分配给一个廉价、低延迟的副模型去执行。通过这种职责分离，系统能够在保持主模型发挥其强项的同时，大幅削减 Token 支出，使得整个 Agent 运行的经济性显著提升。Hermes Agent 官方实践表明，将所有副模型任务指定为 Gemini 1.5 Flash 这样的低成本模型，可以在几乎不影响整体性能的条件下，将 Token 消耗降低数十倍。辅助任务的设计体现了典型的工程权衡思想：不是所有问题都需要最强的工具，合理分层才能实现最优资源配比。

## 技术细节

辅助任务在 Hermes Agent 中通过以下机制实现：

- **任务分类与路由**：系统内置一套启发式规则或轻量级分类器，在接收到用户请求后，初步判断该任务是否属于预设的辅助任务类别（如内容压缩、记忆刷新、MCP 调用等）。若是，则直接将请求转发至副模型，而主模型不被唤醒。
- **副模型配置**：辅助任务的执行模型在配置文件中指定。典型做法是在 `config.yaml` 中设置 `sub_model: gemini-1.5-flash`，并可以为不同辅助任务类型指定不同子模型。
- **状态同步**：副模型执行完成后，其结果需回传给主模型或系统状态管理器，确保主模型在后续复杂推理中能获取到最新的辅助信息（例如压缩后的上下文、记忆向量等）。Hermes Agent 使用统一的中间缓存层来保证数据一致性。
- **安全与审批**：部分辅助任务（如任务批准）需要对人机交互进行确认，副模型仅负责生成待审批的摘要，最终审批动作由用户或预置规则触发，而非副模型自主决策。

## 应用场景

- **长对话记忆刷新**：在持续数小时的会话中，每轮完整上下文会快速消耗 Token。辅助任务定期将历史对话压缩为摘要，并写入记忆存储，主模型只需加载压缩后的语义摘要，极大节省 Token。
- **MCP 工具批量调用**：当 Agent 需要查询多个外部数据源（如天气、股票、日程）时，每个查询本身逻辑简单，可由副模型并行发起调用，再将结果合并返回给主模型做最终分析。
- **内容安全预审**：用户输入中可能包含违规内容，辅助任务先对输入进行关键词匹配和简单语义过滤，只有通过预审的内容才进入主模型处理流程，降低主模型被注入有害指令的风险。
- **视觉轻量处理**：对于图像中的文字提取、基础物体识别等任务，副模型可快速给出结构化输出，主模型仅需基于这些输出进行更高层的推理。
