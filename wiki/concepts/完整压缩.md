## 完整压缩

完整压缩（Full Compression）是四层防御系统中的第三层，也是最深、最昂贵但效果最彻底的上下文压缩策略。当微压缩和会话记忆压缩均不足以把 Token 用量控制在安全阈值以内时，系统会主动触发一次真实的大模型调用，对整个对话上下文进行“思维链引导 + 精炼摘要 + 缓存重置”的全流程重写。完整压缩的核心在于借助大模型自身的推理能力，对历史消息进行语义压缩，同时通过草稿纸模式（Draft Paper Mode）和战后重建机制，防止压缩过程破坏模型的工作记忆和工具调用能力。

完整压缩的设计源于对上下文窗口极限的警惕——当当前 Token 用量超过（上下文窗口上限 − 13,000 Token 安全缓冲）时，系统自动启动这层压缩。与下两层的零成本或低成本不同，完整压缩需要一次完整的模型推理，因此其触发频次被严格控制，通常只在高负载对话或长时间任务中才被调用。

### 技术细节

完整压缩的执行流程包含三个关键步骤：

1. **内置思维链（CoT）：** 系统在 Prompt 中强制要求模型先在一个 `<analysis>` 标签内完整复盘对话的前因后果，包括问题的演进、工具调用的顺序、关键决策点等信息。这一阶段不设 Token 限制，旨在让模型充分理解上下文逻辑。随后模型在 `<summary>` 标签中输出最终摘要。该摘要使用高度凝练的自然语言，仅保留对后续任务有直接帮助的结论、代码片段、配置参数和计划状态。

2. **草稿纸模式：** 系统在拼接最终上下文时，会自动剔除所有 `<analysis>` 段落（仅用于引导推理，不会写入上下文缓存），只保留 `<summary>` 标签中的精炼摘要。此举可在不损失推理质量的前提下，将同一段历史所需 Token 数量压缩至原来的 10%~20%。

3. **战后重建：** 压缩完成后，系统立即清空当前缓存，并重新缝合以下关键内容：
   - 用户当前正在查看的文件（最多 5 个，保持最新状态）；
   - 项目计划与任务列表（包括 MCP 工具配置和 `memory` 记录）；
   - 外挂的 MCP 工具说明（确保工具调用接口可用）。
   这一步骤防止模型因过度压缩而出现“失忆”现象，尤其是丢失对工具调用格式的掌握。

完整压缩还依赖前述两层防御中的结构边界锁和流式碎片绑定来保证数据完整——在压缩过程中，任何工具调用消息与其结果必须成对存在，同时同一条消息内的思考过程与工具指令不得被切断。

### 应用场景

- **长时编程任务：** 当 [[Claude Code]] 在执行连续多个小时的代码生成、重构或调试时，对话历史可能累积数万 Tokens。完整压缩能在保留关键上下文的前提下，将历史对话重构为一份结构化的任务笔记，使模型从全新但富有信息的状态继续工作。
- **多工具链集成：** 在涉及多个 MCP 工具（如数据库查询、API 调用、文件操作）的复杂场景中，完整压缩通过战后重建机制确保所有工具描述和调用历史被正确恢复，避免因缓存变化导致工具失效。
- **大规模文档分析：** 当处理超长文档或大量网页内容时，完整压缩的思维链引导能帮助模型区分重要信息与冗余细节，生成包含关键结论和引用的摘要，极大降低后续对话的 Tokens 消耗。

完整压缩是 [[上下文工程]] 中“以计算换空间”的典型实践，与 [[四层防御系统]] 中的其他三层（微压缩、会话记忆压缩、自动触发与熔断机制）协同工作，构成 Claude Code 稳定运行的关键保障。
