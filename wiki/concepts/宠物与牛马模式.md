# 宠物与牛马模式

**宠物与牛马模式**是 Anthropic 在 **Managed Agents** 架构中提出的核心理念，用于描述 Agent 从“需精心维护的个体”向“可随时销毁重建的无状态工蜂”的范式转变。这一比喻直接指向 Agent 系统在大规模并发场景下的稳定性与可恢复性问题。

## 概念来源与背景

在传统 Agent 开发中，开发者习惯将模型的记忆（上下文窗口）与本地执行环境紧密绑定，为模型编写大量修补代码（Harness）以弥补其短时记忆缺陷。然而，一旦模型升级（例如从 Claude 3 到 Claude 3.5），这些专门编写的“补丁”代码迅速失去价值，系统也变得极其脆弱。Anthropic 工程师曾因此浪费数千行代码。这种模式被形象地称为 **“宠物模式”**——你需要每天喂食、洗澡、带它散步，一旦生病整个系统就停摆。

与之相对，**“牛马模式”** 要求 Agent 彻底无状态化：大脑（大模型）只负责发指令，底层的“双手”（执行沙盒）是独立的、可销毁的环境。当沙盒内的代码跑崩、配置冲突或状态污染时，系统不做任何修复尝试，而是直接销毁该沙盒并重新拉起一个干净的实例。整个 Agent 蜂群如同牧场里的牛马——生病了就换一头，群体效率不受影响。

## 技术实现细节

在 Managed Agents 的四层解耦架构中，宠物与牛马模式逐层落地：

1. **第一层（沙盒层）**：将执行环境容器化（Docker 或毫秒级启动的 Cube Sandbox），确保每个执行实例完全独立。一旦出现异常，`Coordinator` 直接销毁容器，不保留任何本地状态。

2. **第二层（会话层）**：Agent 定义（静态模板）与执行过程（动态 Session）分离。一个模板可对应多个独立 Session，每个 Session 不携带历史环境状态，仅通过外部 [[Session Store]] 获取必要上下文。

3. **第三层（上下文管理层）**：记忆不再附着在 Agent 进程上，而是作为外部挂载。通过 `session store` 实现上下文的 Fork、回滚与克隆。Agent 可随时被销毁，其状态由外部存储负责持久化。

4. **第四层（编排器层）**：`Coordinator` 作为不干活的指挥官，负责维护任务图、派发子任务，并在任务执行中管理上下文压缩。由于 Worker 是无状态的，Coordinator 可以自由调度任意沙盒执行任意子任务而无需顾虑残余副作用。

## 关键收益与案例

- **稳定性**：单个沙盒崩溃不会污染其他沙盒或 Coordinator。在多 Agent 蜂群协作中，即使某个 Worker 节点因代码错误或库冲突而失效，系统只需数毫秒即可用新沙盒替换，整体任务不受影响。
- **可伸缩性**：沙盒可以动态创建与销毁，计算资源按需分配。未来用户只需通过手机或眼镜发出指令，云端即可瞬间创建数十个沙盒并行处理，任务结束后全部销毁，计费模式从“按实例时间”转向“按任务完成度”。
- **可复现性**：由于每个沙盒从标准镜像启动，开发者可以精确复现任何执行问题，无需担心“本地环境与生产环境不一致”。

这一模式直接呼应了分布式系统中的经典原则：将节点视为“牛马”而非“宠物”。在 Agent 领域，它被进一步引申为 [[无状态化设计]] 的实践范式，并催生了 [[Agent沙盒]] 与 [[Session Store]] 等关键组件。未来，随着 [[多Agent协作]] 走向企业级基础设施，宠物与牛马模式将成为评估 Agent 系统健壮性的核心指标之一。

> **关键引言**：“要把 Agent 当牛马去用……哪怕某个沙盒代码跑崩了，配置冲突了，没关系直接干掉，重新拉起一个干净的沙盒接着干。” —— 摘自 Anthropic 技术分享
