# 上下文工程

**定义**
上下文工程（Context Engineering）是 AI 工程化第二阶段的核心，关注模型外部信息的管理：如何检索、筛选、组织、压缩并动态注入上下文，使模型在有限的上下文窗口内获得完成任务所需的最相关、最新信息。其本质是“信息设计”。

**在本库的具体例子**
- 在 [[Harness-Engineering-深度简报-构建稳定-可落地的-AI-Agent系统]] 中，上下文工程被类比为「给员工准备好客户资料、产品报价等背景文件」。视频指出，当 Agent 失败时，常见原因不是模型不好，而是上下文欠缺或污染。
- OpenAI 的 [[目录化管理]] 是上下文工程的典型实践：将大量文档拆分为索引和子文档，Agent 按需加载，而不是将所有内容塞入一个 `.md` 文件。这解决了上下文窗口稀缺问题。

**技术实现细节**
1. **检索增强生成（RAG）**：基于向量相似度从知识库中检索相关片段，组成动态上下文。
2. **上下文压缩**：通过摘要或筛选只保留关键信息，去除冗余，减少 token 消耗。
3. **分阶段注入**：先注入全局框架（如任务步骤），当 Agent 需要特定细节时再注入子文档，例如 OpenAI 的 Skills 目录。

**与近似概念的边界**
- **对比 [[提示工程]]**：提示工程决定“怎么说”，上下文工程决定“给什么”。前者是语言设计，后者是信息供给。两者协同工作：Prompt 中包含从上下文中提取的关键点。
- **对比 [[约束工程]]**：上下文工程只关注输入信息质量，不涉及模型执行后的校验、恢复等机制。约束工程包含上下文工程作为其信息边界层的一部分。
