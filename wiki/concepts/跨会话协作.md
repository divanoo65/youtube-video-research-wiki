# 跨会话协作
---

## 概念定义

跨会话协作（Cross-Session Collaboration）是指在人工智能辅助开发和交互过程中，将不同会话（Session）之间的上下文信息进行持久化、结构化传递并延续执行意图的机制。在单个会话中，AI模型能够记忆当前对话的上下文，但一旦会话结束或切换，先前的上下文便会丢失，导致重复解释需求、丢失决策记录、无法延续工作计划等问题。跨会话协作通过专门设计的技能包（如 `handoff`）将当前会话的关键信息——包括详细总结、工作目录、已达成的决策、待办计划以及未来意图——打包为可读性强的 Markdown 文档，供后续会话或不同智能体读取与执行。该机制使得人类开发者与AI之间、多个AI智能体之间的工作流能够跨越时间边界，实现渐进式的系统开发、需求迭代和架构验证。它不仅仅是一个简单的信息归档，更是一种将逻辑讨论“实例化”为可运行原型、通过多方案变体进行发散与收敛的协作范式，从而克服纯文本推理的歧义性，提升复杂项目的协作效率与可验证性。跨会话协作的核心理念在于，每个会话不仅产生输出，更应产出一个可供后续会话独立理解的“上下文包裹”，使得整个开发过程可以像接力赛一样被拆解、递进和复用。

## 技术细节

跨会话协作的实现依赖于以下关键技术组件与流程：

- **`handoff` 技能包**：核心的上下文桥接工具。它能够自动将当前会话的内容归纳为 `handoff-xxx.md` 文件，包含会话总结、工作目录、决策记录、后续计划以及通过参数传递的“未来意图”。该文档采用 Markdown 格式，相比 Claude Code 默认的 JSONL 格式具有更高的可读性，便于人类和AI共同理解。同时，`handoff` 具备去重机制，可以智能过滤已存在于其他 Artifacts（如 PRD、ADR、Commits）中的重复信息。

- **`otype` 工作流**：一种基于原型驱动的跨会话协作范例。其完整流程包括：  
  1. **需求澄清**：明确需要改造的具体模块。  
  2. **痛点定位**：识别用户不愿订阅的原因（如“不清楚能获得什么”）。  
  3. **变体生成**：自动构建包含不同功能重心（如增加“最近文章预览”）的多个原型。  
  4. **本地部署**：在本地 3000 端口启动开发服务器，允许用户通过参数选择并评估不同方案。  
  该工作流使得逻辑层面的讨论能够被实例化为可交互的终端应用，从而在会话间实现方案的发散与收敛。

- **文档结构与标准化**：跨会话协作依赖统一的文档命名规范（如 `handoff-xxx.md`）和内容模板，确保不同会话生成的上下文包裹可以被后续会话自动解析与引用。文档内容通常包括元数据、状态标记、依赖关系等，以支持自动化工具的读取。

## 应用场景

跨会话协作在以下场景中具有显著价值：

- **大型项目的分阶段开发**：当项目需要多个会话分别处理不同模块时，前一会话的决策和实现细节可以通过 `handoff` 传递给下一会话，避免重复劳动和沟通成本。
- **多智能体协作**：在由多个AI角色（如产品经理、架构师、开发者）协同工作的流程中，每个角色可以独立生成上下文文档，并在切换角色时无缝衔接。
- **原型验证与迭代**：通过 `otype` 工作流，开发者可以在不同会话中生成针对同一需求的不同变体原型（如增加或减少某个功能），并在本地部署后根据用户反馈进行迭代，实现“选择而非生成”的高效协作模式。
- **远程或异步协作**：团队成员无法同步工作时，跨会话协作文档可作为知识传递的载体，使任何人都能在当前的进度上继续工作。
