## 概念定义

健康检查（Health Check）是系统可观测性中的核心机制，用于对基础设施、服务依赖或关键配置进行持续性、低成本的存活状态验证。在 Hermes Agent 网页版中控台的具体实践中，健康检查被抽象为一种“一眼识别”的视觉化状态屏：API Key 的健康度以绿色表示正常、红色表示缺失或失效；服务网关（如 Telegram Gateway）的运行状态也通过同样的红绿灯逻辑展示。这种设计将复杂的后台探测逻辑转化为直觉化的界面反馈，使运维人员或普通用户能在不查阅日志的情况下，快速定位故障根因。健康检查不仅涵盖网络连通性，还延伸到凭据有效性、配额余量、服务响应时延等维度。从系统设计角度看，健康检查是“防御性监控”的起点——它不要求精确的故障诊断，而是提供“有无故障”的二进制信号，进而触发自动恢复流程或告警通知。在 Hermes 的架构中，健康检查与成本管理（费用拆解）并列于“成本与资源管理”面板，表明其定位不仅是技术可用性指标，更是资源健康度的经济性视图：一个 Key 的失效可能直接导致调用失败并产生额外重试费用。因此，健康检查充当了从技术状态到商业影响的桥梁，是 AI Agent 系统中不可或缺的可靠性基石。

## 技术细节

健康检查的实现通常依赖以下机制：

- **探测频率与超时**：后台定时任务（如每30秒）向目标服务发送轻量级请求（如 HTTP HEAD 或 TCP ping），设置较短超时（如3秒）以避免阻塞。
- **状态机模型**：从“健康”（Healthy）到“不健康”（Unhealthy）的转换需经过多次连续失败（例如3次）以防止抖动；恢复时同样需要连续成功若干次。
- **可视化映射**：前端通过 WebSocket 或轮询获取后端状态数据，将布尔值渲染为绿（正常）或红（异常）指示灯，有时辅以黄色表示降级。
- **依赖链检查**：在 Hermes 中，API Key 的失效可能连带影响所有依赖该 Key 的模型调用，因此健康检查会展示“网关→Key→模型”的层级关系。
- **历史记录**：存储每次状态变更的时间戳，供审计和根因分析使用。

## 应用场景

健康检查在以下场景中发挥关键作用：

1. **API 凭据管理**：中控台集中管理多个第三方 API Key（如 OpenAI、Claude），健康检查能实时反映哪个 Key 已过期或余额不足，避免调用时返回 401/403 错误。
2. **服务网关监控**：Telegram Gateway 等入口服务的可用性直接影响用户消息的送达率，健康检查面板可提前预警通信链路中断。
3. **自动化运维**：结合 [[Hermes Agent 网页版中控台深度分析报告]] 中提到的 Cron 任务，健康检查可触发自动化恢复脚本（如自动刷新过期 Key 或切换备用网关）。
4. **项目健康总览**：在 [[项目扫描]] 中，健康检查与 Git 状态（Active/Dirty/Stale）共同构成项目级健康视图，帮助开发者判断当前工作环境的完整性。
