# 宠物模式

## 概念定义

宠物模式（Pet Mode）是 AI Agent 系统设计中的一种典型反模式，指将 Agent 的“大脑”（大语言模型、规划逻辑）与“双手”（执行环境、本地资源、记忆存储）进行深度耦合的开发方式。在这种模式下，每个 Agent 都像一只需要精心照料的宠物：它拥有独特的配置、累积的历史记忆、依赖特定的操作系统或库版本，甚至需要定期“投喂”（手动调试）才能维持正常运作。一旦环境发生变化（例如依赖库升级、文件路径迁移、系统更新），Agent 就可能因配置冲突、记忆冗余或环境污染而崩溃，且难以恢复。传统上，开发者为每个 Agent 单独搭建环境、手工调整参数，导致系统整体缺乏弹性，部署和扩缩容成本极高。宠物模式的核心问题在于将状态（记忆、环境）与执行体绑定，违背了现代基础设施的无状态化、可替换性原则。Anthropic 在分析 Managed Agents 架构时明确指出，这种模式必须被淘汰，转向将 Agent 视为基础设施中的“牛马” —— 无状态、高并发、随时可销毁重建。

## 技术细节

宠物模式的技术特征主要体现在以下几个层面：

1. **状态耦合**：Agent 的对话记忆、工具调用历史、环境变量等状态被持久化在本地文件系统或数据库，并且与特定的虚拟机或容器实例绑定。一旦实例宕机，状态丢失或持久化路径失效，Agent 便无法恢复工作。
2. **环境依赖**：Agent 的执行脚本依赖于安装好的特定 Python 包、Node.js 版本、系统库等，且这些依赖常与操作系统版本、网络配置耦合，导致环境漂移难以追踪。每次启动 Agent 都可能因依赖冲突报错，需要手动修复。
3. **单点故障**：由于每个 Agent 拥有独立的环境和记忆，其生命周期与宿主机紧密绑定。宿主机故障（如磁盘空间不足、网络中断）直接导致 Agent 不可用，且无法通过简单的重部署恢复——必须手动重建环境、迁移记忆。
4. **扩展困难**：若需要增加 Agent 数量，必须为每个新 Agent 重复环境配置、记忆导入等过程，无法通过水平扩展自动完成。这在多 Agent 协同（如“蜂群模式”）场景下成为瓶颈。

Anthropic 提出的“牛马模式”将大脑（LLM + 规划器）与双手（基于 Docker 或 Cube Sandbox 的无状态执行沙盒）彻底解耦。大脑只发出指令，底层沙盒会被频繁销毁重建，所有状态外移至独立的会话存储层（如云端 Agent 操作系统中的记忆树）。这种解耦彻底消除了宠物模式的痛点。

## 应用场景

- **传统代码辅助 Agent**：早期基于 LangChain 搭建的 Agent 常将对话历史、文档索引直接存储在本地目录，更换服务器后需全量迁移，属于典型的宠物模式。
- **企业级 RAG 系统**：一些 RAG 应用将向量数据库嵌入 Agent 进程内，导致 Agent 重启后需要重新构建索引，无法快速故障转移。
- **个人 AI 助手**：将用户偏好、日历数据等长期记忆硬编码在 Agent 的环境变量中，难以跨设备同步，也无法实现多实例间共享。
- **开源 Agent 框架**：部分框架要求用户在每台宿主机上手动安装运行时、配置 API Key 并启动 Agent，无任何容器化或编排机制，本质是宠物模式。

了解宠物模式有助于开发者识别系统中潜在的脆弱点，并参考 [[牛马模式]] 与 [[脑手分离]] 原则进行重构，从而实现更健壮的 [[托管代理架构]]。
