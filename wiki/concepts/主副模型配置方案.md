# 主副模型配置方案

## 概念定义

主副模型配置方案是 Hermes Agent 为应对大语言模型 API 调用成本过高而设计的一种智能分工机制。该方案的核心思想在于：将任务按照复杂度与推理需求进行拆解，由昂贵的高性能“主模型”专门处理复杂推理、核心指令执行等高价值任务，而将辅助性、重复性或计算量较小的“副模型”任务（如任务批准、内容压缩、记忆管理、工具调用、多模态解析等）交由成本极低的小模型（例如 Gemini 1.5 Flash）来承担。通过这种“强模型把关关键逻辑，弱模型处理外围琐事”的架构，用户可以在几乎不牺牲核心推理质量的前提下，将整体的 Token 消耗降低 50% 以上，特别适合需要长期运行、高频调用 API 的自动化 Agent 场景。该方案不仅适用于云端模型，同样可以结合本地部署的轻量模型（如 Ollama 上的小参数模型）进一步优化成本结构，使得个人开发者也能负担起复杂的 AI Agent 持续运行。

## 技术细节

### 任务分工表

| 任务类型      | 建议模型级别      | 具体应用场景                         |
| ------------- | ----------------- | ------------------------------------ |
| 主任务        | 昂贵/高性能模型   | 复杂推理、核心指令执行、代码生成     |
| 辅助任务      | 廉价/副模型       | 任务批准（Approval）、内容压缩（Compression） |
| 记忆管理      | 廉价/副模型       | 刷新记忆、会话搜索、历史摘要         |
| 工具调用      | 廉价/副模型       | MCP 调用、Skill 相关任务、API 编排   |
| 多模态/网页   | 廉价/副模型       | 视觉解析（Vision）、网页内容提取     |

### 配置实现

在 Hermes Agent 的配置文件中，用户需要显式启用主副模型开关，并分别为两类任务指定不同的模型端点与认证信息。典型的配置项包括：

- `PRIMARY_MODEL`：指定主模型（如 GPT-4、Claude Opus）
- `SECONDARY_MODEL`：指定副模型（如 Gemini 1.5 Flash、Llama 3.2 8B）
- `TASK_ROUTING`：定义任务类型到模型类别的映射规则
- `ENABLE_API=true`：启用 API 服务以支持跨平台访问
- 认证密码设置用于保护副模型调用的安全

### 跨平台与会话管理

副模型不仅负责 Token 节省，还承担了 [[API服务]] 的底层支持，例如通过 IP 与端口（默认 8080）提供手机浏览器访问功能，允许上传文件、截图和引用网页。历史会话保存于左侧侧边栏，并支持全局搜索，这些由副模型完成的记忆管理任务极大减轻了主模型的负担。

## 应用场景

- **长期运行的个人 AI 助理**：用户需要 Agent 持续监听聊天、处理文件，如果所有交互都使用 GPT-4，每月 Token 费用将十分高昂。采用主副模型方案后，日常对话摘要、消息分类、任务确认等均交给副模型，只有遇到复杂指令时才触发主模型，成本可降低 70% 以上。
- **企业级自动化工作流**：在需要频繁调用 MCP 工具、访问数据库、生成周报的系统中，通过副模型处理反复性的工具调用和结果格式化，主模型仅负责异常处理与策略决策，显著提升吞吐量并控制预算。
- **移动端与低算力设备**：副模型可部署在本地（如 [[Ollama]] 上的小模型），主模型使用云端高性能模型，从而在不依赖稳定网络的情况下完成基本功能，同时保留复杂任务时连接云端的能力。
- **教育与研究**：研究人员可以用低成本副模型进行大规模实验筛选，仅对最有希望的方案动用主模型做精细分析，极大加速研究迭代周期。
