# 微压缩

## 概念定义

微压缩（Micro-Compression）是 [[Claude Code]] 上下文工程架构中四层防御系统的基础层，位于整个压缩策略的最前端。其核心设计哲学是“零成本的规则清理”：绝不调用大模型，完全基于预设的规则引擎对上下文内容进行快速筛选与裁剪。微压缩主要针对那些高频出现但时效性极短的工具输出数据，例如文件读取结果、grep 脚本输出、网络搜索返回的片段等。系统仅保留最近的几个结果，并清理早期产生的冗余信息，从而在几乎不消耗计算资源的前提下有效控制上下文膨胀。由于不涉及模型推理，微压缩的执行速度极快，且对 API 缓存友好，能为后续更重量的压缩层（如[[会话记忆压缩]]、完整压缩）减轻负担。微压缩的另一个关键特性是“线程隔离”——不同对话线程或子任务中的微压缩操作互不干扰，确保每个独立工作流的上下文清洁度。

## 技术细节

微压缩的实现依赖于两套并行的决策路径，根据对话连接状态自动切换，具体如下：

- **在线路径（Catch Microcompact）**：当用户持续高频交互时，服务端的 Anthropic API 缓存仍处于有效期。系统通过 API 发送指令，直接从上下文列表中移除旧的工具输出结果，但不改动原始 Prompt 前缀（即系统提示和初始对话片段）。这样既能缩小上下文长度，又能保持缓存命中率，兼顾成本与速度。
- **离线路径（HB Microcomplex）**：如果对话中断时间较长（例如用户短暂离开后返回），API 缓存已过期。此时系统无法通过 API 指令进行原位删除，而是在本地将冗长的工具输出替换为极短的结构化占位符（如 `[Omitted tool output: read_file 3 results]`），待后续需要时再通过 [[按需加载工具]] 恢复。这种本地替换不依赖网络，也不影响缓存的重新建立。
- **白名单机制**：微压缩仅作用于预先定义的白名单工具（如 `Read`, `Grep`, `WebSearch` 等），对模型自主生成的思考链、用户消息等关键内容不做压缩，避免误伤重要信息。
- **滑动窗口保留**：每个白名单工具的输出按时间顺序排列，系统维护一个固定大小的滑动窗口（通常保留最近 2-3 条结果），超出窗口的部分被立即标记为可移除。窗口大小可通过配置调整，以平衡信息完整度与压缩率。

## 应用场景

微压缩在以下场景中发挥关键作用：

1. **大规模代码库探索**：当 Claude Code 连续执行多轮文件读取、目录遍历或代码搜索时，早期的输出迅速失去参考价值。微压缩自动清理过时的文件内容，防止上下文被重复信息填充，从而为后续模型分析[[熔断机制]]和深度推理留出空间。
2. **高频任务循环**：在自动化脚本执行或增量构建过程中，工具调用密集且结果往往仅对当轮有用。微压缩确保每个循环周期开始时上下文都是“轻量”的，避免累积历史垃圾导致 token 溢出。
3. **长对话中的早期干预**：对话早期一旦出现大量工具输出，微压缩可立即介入，无需等待上下文达到临界值才触发更重的压缩策略，实现“防患于未然”。
4. **配合[[双轨决策机制]]成本优化**：在线路径下的微压缩充分利用缓存优势，使得频繁的上下文清理几乎不产生额外 API 费用，成为整个压缩体系中最经济的环节。

通过微压缩，[[Claude Code]] 能够在数百轮对话和数万行源码的交互中持续维持高效的上下文利用率，将“昂贵且稀缺的战略资源”留给真正需要模型深度理解的逻辑推理与决策过程。
