## 定义

健康检查（Health Check）是 [[llm-wiki]] 体系中的一项关键运维机制，通过定期执行自动化脚本（类比代码审计），对知识库进行全域扫描，主动发现并修复内容矛盾、过时信息、孤立页面和缺失引用等问题，从而持续维护知识库的有序性与可靠性。这一概念突破了传统知识管理“建完即弃”的静态维护模式，将知识库视为一个需要持续养护的有机体，确保其随着时间推移和内容增长仍然保持结构完整、逻辑自洽。健康检查的核心价值在于打破“数字垃圾”积累的恶性循环——当笔记数量膨胀至数千篇时，人工手动维护几乎不可能；而自动化检查能够以极低成本实现定期的“知识体检”，让知识库始终处于接近生产环境可用的高质量状态。在 LLM-Wiki 的三层架构中，健康检查位于持久层之上，与摄入和查询与回填共同构成闭环；它既是对前两者产生内容的质检，也是后续 [[query-and-refinement]] 动作准确性的保障。

## 技术细节

健康检查的实现通常遵循以下流程：

1. **扫描范围界定**：通过 Markdown 索引（如 `Index.md`）或 [[dataview]] 查询获取所有知识库文件的路径与元数据。利用文件系统遍历或 Obsidian 插件 API 收集页面列表。
2. **规则引擎执行**：预定义一组检查规则，例如：
   - **孤立页面检测**：检查是否存在没有任何其他页面引用的页面（即入度为零）。孤岛页面会降低知识网络的连通性。
   - **内容矛盾检测**：对具有相同或相近标题的页面进行语义相似度对比（可使用小型语言模型或关键词碰撞），标记明显冲突的陈述。
   - **时效性检查**：根据每页的 `updated` 字段与当前时间差，标记超过一定阈值（如6个月）未更新的页面，提示可能过时。
   - **缺失引用检查**：扫描文中的 `wikilink` 语法，验证目标页面是否存在。不存在的链接会被记录为断链。
   - **格式完整性检查**：确保每个页面都包含必要的 frontmatter 字段，避免因元数据缺失导致查询异常。
3. **报告生成与修复**：每次健康检查生成一份 JSON 或 Markdown 报告，列出所有问题及其严重程度。对于可自动修复的问题（如补全缺失的 frontmatter、删除指向不存在页面的链接），脚本直接执行修正；对于需要人工判断的问题（如内容矛盾），则在对应页面添加 `⚠️ 健康检查提示` 的备注块，并同步至修订待办列表。

在工具链层面，健康检查通常依托 [[vs-code]] 的 Task Runner 或 Obsidian 的 Templater + QuickAdd 组合，以定时任务（Cron Job 或 GitHub Actions）形式每日/每周运行。LLM-Wiki 的高级版本会引入大模型辅助语义审查，例如使用 [[claude]] 或 [[deepseek]] 对疑似矛盾片段进行二轮判断，减少误报。

## 应用场景

- **个人知识库的持续迭代**：对于长期积累了上千条笔记的 Obsidian 用户，健康检查可以每周自动发现因重构、移动文件而产生的孤立页面和断链，避免知识网络退化。例如，当用户将一个长篇笔记拆分为多篇主题页面后，健康检查能自动发现并建议补充跨页面的 [[bi-directional-link]]。
- **团队协作的知识管理**：在多作者编写的 LLM-Wiki 中，健康检查可以统一不同成员的写作规范，发现重复或冲突的内容片段。例如，两个成员分别撰写了“RAG 优化”和“检索增强生成调优”，健康检查会标注二者的相似性并建议合并。
- **知识库的“编译”验证**：借鉴编程中的持续集成（CI）思想，在每次通过 [[obsidian-web-clipper]] 摄入新内容后立即触发健康检查，确保新增页面不会破坏既有结构的完整性。类似 [[compiler-pattern]] 对代码的静态检查，健康检查是对知识图谱的静态分析。

健康检查与 [[query-and-refinement]] 构成知识库的自我修复闭环：当日志记录了修复动作后，后续的查询就能基于更干净的数据给出更精确的回答，从而实现 [[knowledge-compounding]] 的指数级增长。
