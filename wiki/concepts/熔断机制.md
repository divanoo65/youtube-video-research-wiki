## 概念定义

熔断机制是 Claude Code 上下文工程架构（[[四层防御系统]]）中位于第四层的关键保护策略，其核心目的是在对话或任务执行过程中，当上下文资源（Token 容量、模型推理时间、缓存命中率等）逼近预设阈值或出现异常增长时，由系统自动触发中断操作，强制停止当前正在进行的处理流程，并切换至降级或恢复模式。与简单的超时中断不同，熔断机制具有“先诊断后熔断”的特点：系统会持续监控多个指标（如累积工具输出长度、用户输入频率、模型响应时间），一旦检测到某个指标超过安全边界（例如工具输出列表膨胀导致上下文爆满），便会立即发出熔断信号，阻止后续请求进入模型，同时清理已占用的临时资源，并启动[[自动触发]]的“战后重建”流程，将重要信息压缩、转移至持久化存储（如[[会话记忆压缩]]后的结构化摘要）。该机制借鉴了分布式系统中的断路器模式，将 AI Agent 的上下文视为不可再生的战略资源，通过熔断避免因资源耗尽导致整个对话的“思维断层”或模型陷入无限重复的垃圾输出。在 Claude Code 中，熔断机制与[[微压缩]]、[[完整压缩]]配合使用，形成了“预防-诊断-熔断-恢复”的闭环，确保 Agent 在数百轮对话和数万行源码的复杂任务中始终保持逻辑一致性和运行效率。

## 技术细节

熔断机制的技术实现可分为三个阶段：

1. **监控与阈值定义**：系统持续追踪四类关键指标——上下文 Token 使用率（接近 90% 时触发预警）、无新信息增量的重复循环次数（如同一工具调用超过 5 次）、模型响应延迟的移动平均偏差（超过正常值 3 倍标准差），以及工具输出总量的增长速度（如单次文件读取超出上下文容量的 20%）。这些指标通过流式碎片绑定（[[流式碎片绑定]]）实时采集，不阻塞主线程。

2. **熔断决策**：当任意指标超过刻熔断阈值时，系统进入“预熔断”状态，此时不再接受新的工具调用或用户输入，而是触发一个独立的诊断模型对当前上下文进行快速评估。诊断模型分析“失忆风险”（如关键变量被覆盖）、“上下文污染”（如无用日志占据大量空间）和“缓存失效概率”（如 [[Anthropic]] API 缓存因前缀变化而失效），并输出一个熔断优先级（0-1）。若优先级 > 0.7，则执行硬熔断——立即取消所有待处理任务，释放 GPU 或 API 连接资源；若在 0.3-0.7 之间，执行软熔断——暂停当前任务，允许用户选择是否继续或回滚。

3. **恢复与补偿**：熔断触发后，系统调用[[指针替换]]机制，将原始长文本替换为简短占位符，同时将完整内容压缩并移至持久化层。随后启动[[战后重建]]流程，通过[[双轨决策机制]]决定是恢复最近检查点还是从结构摘要中重构上下文。整个熔断过程的时间复杂度为 O(1)，不依赖大模型推理，确保熔断本身不会成为资源瓶颈。

## 应用场景

1. **长对话中的资源保护**：当用户与 [[Claude Code]] 进行超过 200 轮的大型代码重构对话时，大量历史文件读取、grep 结果堆积可能导致上下文 Token 溢出。熔断机制会在 Token 使用率达到 85% 时自动触发软熔断，提示用户保存当前进度并压缩历史，避免对话意外中断。

2. **递归工具调用防止死循环**：AI Agent 在调试代码时可能陷入“读取-修改-验证-再读取”的无限循环，每次循环生成新的文件输出，迅速撑爆上下文。熔断机制通过检测工具调用模式的重复性（如连续 5 次调用同一个文件读取 API 且内容不变），立即中断循环并强制切换为不同的分析策略。

3. **多 Agent 协作下的资源争用**：在 [[Claude Code 上下文工程架构深度解析简报]] 描述的线程隔离架构中，主线程与子 Agent 共享有限上下文空间。当子 Agent 输出碎片（如流式碎片）积累过快时，熔断机制根据线程优先级丢弃低优先级子 Agent 的输出，保障核心任务的连续性。

4. **成本控制与缓存优化**：由于 Anthropic API 缓存依赖于稳定前缀，一旦熔断触发后的恢复流程改变了 prompt 前缀，缓存可能失效。熔断机制会优先选择不改变前缀的压缩方式（如[[微压缩]]），仅在极端情况下才执行全量重构，从而平衡熔断的成本与收益。
