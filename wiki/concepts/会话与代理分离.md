## 概念定义

会话与代理分离（Session-Agent Separation）是当代 Agent 系统架构中的一项核心设计范式，其本质是将“用户与系统的一次交互上下文（会话）”与“实际执行任务的代理实例”彻底解耦。在传统单体 Agent 中，代理实例通常绑定一个会话——这意味着每次对话都需要维持一个长期驻留的代理进程，代理状态（记忆、上下文、中间结果）全部附着于该进程；一旦代理重启或失败，会话便丢失，导致用户体验割裂、资源浪费且难以横向扩展。而“会话与代理分离”要求：会话本身是独立持久化的实体，包含完整的对话历史、用户意图、中间变量和任务栈；代理（一个轻量的规划-执行循环）则被设计为完全无状态（Stateless）的“工蜂”，每次接到调度指令时从会话存储中读取最新状态，执行完一个步骤后将结果写回会话，随后自身销毁。这种模式使得一个会话可以被多个代理接力完成，或一个代理被复用于多个会话（在毫秒级沙盒中），从而在根本上解决了长程任务的状态维护、故障恢复与高并发调度难题。Anthropic 在 Managed Agents 架构中明确提出这一分离，并将其与“脑手解耦”、“记忆存储独立管理”共同列为构建企业级Agent操作系统的三大支柱。

## 技术细节

会话与代理分离在实现上依赖三个关键组件：

1. **会话存储层（Session Store）**：独立于代理运行的持久化服务，通常采用高速 KV 存储（如 Redis）或日志型数据库（如 FoundationDB）。每个会话拥有唯一标识（Session ID），存储的数据结构包括：消息队列（用户输入/模型输出）、上下文窗口（Token 序列）、任务 DAG 图（已完成/待执行步骤）、变量快照（Agent 运行时写入的临时结果）。该层必须支持事务性读写，以保证并发安全——因为多个代理可能并行处理同一会话的不同子任务。

2. **无状态代理工厂（Stateless Agent Factory）**：每次需要执行动作时，系统从工厂拉取一个纯净的代理实例（通常运行于沙盒容器内）。该实例从零初始化，不携带任何记忆，只通过环境变量获得当前会话 ID。它从会话存储下载必要的上下文，执行推理与工具调用，然后将更新后的状态回写，最后自行销毁或返回池中。代理的生命周期与调用次数对齐，而非与会话对齐。

3. **调度器（Dispatcher）**：管理会话-代理的关联关系。它根据任务的紧急程度、模型类型、可用资源等调度策略，从工厂获取代理并绑定到特定会话的一个步骤；执行完成后，调度器标记步骤完成，并可能触发下一步或等待外部事件。调度器还负责处理超时重试、错误回滚——由于会话状态独立，失败代理可以直接被替换，无需重建对话。

## 应用场景

- **多轮对话的长程任务**：例如用户要求“分析上季度销售数据并生成PPT”，传统 Agent 必须保持长连接以维持状态。采用会话与代理分离后，第一步数据分析可由一个代理在沙盒中执行 SQL，结果写入会话；第二步生成图表由另一个代理调用绘图工具；第三步合成 PPT 由第三个代理调用 API——每个代理只关注一个步骤，会话状态持久化保证了上下文的连贯传递，而且任一环节失败都可从会话快照恢复。

- **高并发的客户服务系统**：电商客服场景中，大量会话可能同时涌入，每个会话包含多次转人工、转系统逻辑。将代理设计为无状态后，一台物理机可同时运行数百个沙盒容器，每个容器处理一个会话的当前步骤，完成后迅速释放；会话本身存储在分布式缓存中，实现水平伸缩。即使某个容器崩溃，会话数据不丢，立即分配新容器即可继续。

- **多 Agent 协作（蜂群协作）**：在 [[脑手解耦]] 架构下，一个复杂的推理任务（如代码审查、多源信息融合）可能需要多个专业 Agent 协同。会话作为共享黑板（Blackboard），每个 Agent 读取/写入同一会话的不同区域，调度器负责编排顺序。这种模式天然支持分布式并行，且避免了 Agent 间直接通信的耦合。

## 相关概念

- [[无状态化]]：会话与代理分离是“无状态化”原则在 Agent 领域的具体体现，代理本身不持有状态，所有状态外移到会话存储层。
- [[Managed Agents 架构]]：Anthropic 提出的企业级 Agent 架构将该分离作为核心层，与脑手解耦、记忆存储独立管理共同构成四层解耦结构。
