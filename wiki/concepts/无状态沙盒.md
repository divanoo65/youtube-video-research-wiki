# 无状态沙盒

无状态沙盒（Stateless Sandbox）是一种为 AI Agent 设计的运行时隔离机制，其核心思想是将 Agent 的**大脑（模型决策）**与**双手（代码执行环境）**彻底解耦。每个执行任务都在一个独立、一次性、可随时销毁并重建的沙盒环境中运行。一旦沙盒内发生环境污染、状态混乱或执行错误，系统立即销毁该沙盒并快速拉取一个全新的干净副本，从而实现**故障隔离**与**快速重置**。这种设计借鉴了云原生容器的最佳实践，将 Agent 从需要精心呵护的“宠物”转变为可以像“牛马”一样无情替换的劳动力。

## 背景与痛点

在传统 Agent 架构中，模型、上下文记忆与本地执行环境深度绑定。Agent 长期运行时，容易积累冗余记忆、环境变量冲突或依赖版本错乱，最终导致系统崩溃或“变笨”。修复一个带状态的长程 Agent 需要复杂的回滚或诊断，代价极高。无状态沙盒正是为了解决这些痛点而被提出。

## 技术实现细节

根据 [[蜂群思维|蜂群思维：多Agent协作架构的“版本答案”]] 报告中引用的 Anthropic Managed Agents 实践，无状态沙盒的关键设计包括：

- **环境隔离**：每个执行任务启动时，系统为其分配一个独立的沙盒容器（如 Docker 或高性能的 **Cube Sandbox**）。该容器预装了所需的语言运行时、工具链和依赖库，但**不绑定任何持久化状态**。
- **会话与执行分离**：Agent 作为静态模板只定义身份和工具集，而 Session 负责承载动态的对话线程与临时文件操作。同一个 Agent 可以同时拥有多个 Session，每个 Session 拥有独立的沙盒环境，互不干扰。
- **销毁-重建策略**：如果任务执行过程中发生致命错误、环境配置被意外篡改，或任务完成后需要释放资源，系统直接**销毁整个沙盒容器**，并立即从一个干净的基线镜像重新创建。这一过程通常可在**毫秒级**内完成，确保后续任务在纯净环境中运行。
- **上下文云端化**：为了配合无状态沙盒，Session 的上下文（记忆树）被存储到云端（如 Redis 缓存），称为 **Session Store**。这样即使沙盒被销毁，上下文仍然可以从云端 Fork、回滚或克隆，保证了 Agent 在故障恢复后仍能保持“最聪明”的状态。这标志着 [[Session Store]] 作为 Agent 外部存储的重要性。

## 核心优势

1. **故障隔离**：一个沙盒内的崩溃不会影响其他沙盒或编排器主进程。Agent 不再因为某个子工具的超时而整体瘫痪。
2. **快速重置**：恢复一个污染的环境不再是手动打补丁，而是直接销毁重建。这与云原生中的 “Immutable Infrastructure” 理念一脉相承。
3. **并发扩展**：因为沙盒无状态，编排器可以轻松创建成百上千个并行沙盒，每个处理独立子任务（如搜索、编码、测试），实现真正的蜂群并发。[[Agent编排]] 模式中，一个编排者可以管理多达 20 个拥有独立 Prompt 和 MCP 服务器的 Agent，每个 Agent 都在自己的沙盒内运行。
4. **成本优化**：沙盒按需创建，任务完成后立即销毁，避免了长期运行容器的资源浪费。同时，编排器可以根据任务复杂度灵活调用不同模型（大小模型异构组合），进一步提升 Token 效率。

## 实践启示

开发者在构建大规模 Agent 系统时，应放弃将环境状态与 Agent 生命周期绑定的旧思路。无状态沙盒是迈向企业级 Agent 基础设施的核心基石。正如引用报告所述：“你要扩大 Agent 的规模，就要解耦‘脑’和‘手’。” 未来，云端计算资源将不再以长期运行的服务形式存在，而是由无数个毫秒级启动、自动销毁的 Agent 沙盒组成的动态蜂群。
