# 解释器模式

## 概念定义

解释器模式在 LLM-Wiki 的知识管理语境中，被用来描述传统检索增强生成（RAG）系统的工作方式。与计算机科学中经典的「解释器设计模式」不同，这里的含义借用了“解释器”与“编译器”的隐喻：解释器模式下的知识处理具有临时性和一次性的特点。当用户向系统发起查询时，系统会立刻检索相关的原始文本片段，并将这些片段直接拼接到大模型的输入上下文中，由大模型现场“解释”并生成答案。整个过程不产生任何持久化的结构化知识，每一次查询都相当于从零开始，无法对历史查询结果进行积累、复用或迭代。这种模式就像一位不懂记忆的翻译官，每次听到一句话都重新开始理解，不会因为之前翻译过类似句子而变得更熟练。其本质是**线性叠加**：知识库被视为静态的数据切片，新查询与旧查询之间没有关联，更不存在知识网络的生长。因此，它缺乏**复利效应**：第一篇笔记不会让第二篇笔记变得更聪明，系统不会因为使用次数增加而进化。

## 技术细节

- **工作流程**：用户提问 → 向量检索（或关键词检索） → 召回 top-k 文档片段 → 拼接为 Prompt → 大模型生成回答 → 丢弃上下文。整个过程无状态。
- **数据层**：原始数据（论文、网页、笔记）以扁平形态存储，通常使用向量数据库或倒排索引。数据之间没有显式的链接或层级结构。
- **关键局限**：
  - **上下文窗口瓶颈**：每次只能塞入有限数量的片段，大模型无法感知整个知识图谱。
  - **知识碎片化**：同一概念在不同文档中被重复描述，系统无法合并歧义或建立统一理解。
  - **无回写机制**：生成的回答不会反哺知识库，答案质量完全依赖检索精度和模型即时推理能力。
- **与传统设计模式类比**：在 GoF 设计模式中，解释器模式通常用于为特定语言定义语法并解释执行。而此处借用该词，意在强调“边解释边执行、无预先编译”的临时性特征。

## 应用场景

- **简单问答系统**：如早期的客服机器人，用户问一句，系统搜一句，答完即止，无长尾知识积累。
- **一次性文档分析**：用户上传一份 PDF，让大模型总结内容，总结完成后知识不被留存。
- **原型验证**：在快速搭建 PoC（概念验证）时，解释器模式因实现简单而被广泛采用，但长期使用会导致知识管理效率低下。
- **与编译器模式的对比**：LLM-Wiki 体系明确将解释器模式视为需要被替代的旧范式，转而采用 [[compiler-pattern]] 以实现知识的复利增长。同时，该模式也是理解 RAG 系统演进的基础视角。
