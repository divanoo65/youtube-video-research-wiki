# Harness Engineering（约束工程）

**定义**
Harness Engineering（约束工程）是 AI 工程化第三阶段的核心范式，指在模型之外构建一套完整的运行约束系统（Harness），包括信息边界、工具管理、执行编排、记忆状态、评估观测和故障恢复等机制，核心公式为 `Agent = Model + Harness`。其目标不是提升模型能力，而是让模型在真实任务中“不跑偏、跑得稳、可纠偏”。

**在本库的具体例子**
- 在 [[Harness-Engineering-深度简报-构建稳定-可落地的-AI-Agent系统]] 中，提出了六层架构：信息边界层定义“我是谁”和任务成功标准；工具系统层管理工具选择与时机；执行编排层实现目标理解→任务串联→输出检查的闭环；记忆与状态层区分当前任务、中间结果与长期记忆；评估与观测层建立独立验收机制；约束与恢复层提供重试、回滚和路径切换能力。
- Anthropic 的 [[独立评估者模式]] 是 Harness 架构中评估与观测层的具体实现；OpenAI 的 [[目录化管理]] 是信息边界层的优化手段。

**技术实现细节**
1. **信息边界构建**：在系统 Prompt 或结构化配置中明确角色、目标、成功标准，并通过规则引擎动态裁剪外部上下文，避免无关信息污染。
2. **执行编排闭环**：采用有限状态机或工作流框架（如 LangGraph、Temporal）串联原子操作，每个步骤包含输入校验、前置条件检查和输出验证。
3. **故障恢复机制**：维护任务状态的持久化快照，在 API 超时或异常时，从最近的检查点恢复而非从头重试，同时支持备选路径（如降级使用本地模型）。

**与近似概念的边界**
- **对比 [[提示工程]]**：提示工程关心指令的措辞表达，属于“语言设计”；约束工程关心系统整体的运行约束，属于“系统设计”。前者是后者中信息边界层的一部分。
- **对比 [[上下文工程]]**：上下文工程关注给模型提供正确的信息（时机与内容）；约束工程关注模型在拿到信息后如何稳定执行任务、如何回退、如何验证结果。上下文工程是约束工程中信息边界层和工具系统层的基础。
- **对比传统软件工程**：传统软件工程中的约束（如类型检查、断言）是确定性约束，Harness 中的约束需处理概率性模型的不确定性，如重试策略、置信度阈值、人工回退。
