## 概念定义

项目扫描是 Hermes Agent 网页版中控台内嵌的一项智能检测功能，旨在替代开发者手动检查项目仓库状态的低效操作。它通过后台持续扫描本地文件系统中预先注册或自动发现的项目目录，实时获取每个项目的 Git 存储库状态，并以可视化的方式呈现在中控台界面中。该功能解决了多项目并行开发时，开发者需要频繁切换终端窗口、逐一执行 `git status` 的痛点，将原本分散的仓库状态信息集中为一个统一的看板。项目扫描不仅能够识别仓库是否处于“干净”状态，还能标记出尚未提交的修改（Dirty）、长期未更新的陈旧项目（Stale）以及当前正在活跃开发的分支（Active）。这一能力与中控台整体的“自动化技能”理念一脉相承——将重复的手动操作转化为系统自动完成的流程，使开发者能够将注意力集中在更具创造性的编码任务上。此外，项目扫描的结果可以直接触发后续动作，例如当检测到某个仓库进入 Dirty 状态时，系统可以自动建议生成提交摘要或创建定时备份任务。

## 技术细节

项目扫描的核心实现依赖于 Hermes Agent 后端的文件系统监听模块以及 Git 命令的封装。该模块会维护一个项目路径列表（可通过 Web UI 手动添加或从工作目录自动发现），每隔一定时间间隔（例如 5 分钟）对每个路径执行 `git status --porcelain` 等轻量级命令，并解析输出以判断状态。扫描结果被持久化存储在 `state.db`（本地 SQLite 数据库）中，供前端通过 REST API 或 WebSocket 实时拉取。其关键指标包括：

- **Active**：最后提交时间在 7 天以内，且工作区与暂存区均无未提交变更。
- **Dirty**：工作区或暂存区存在未提交的变更（包括新增、修改、删除文件）。
- **Stale**：最后提交时间超过 30 天，且无活跃分支操作。

为了避免频繁 I/O 对开发者主机性能造成影响，扫描模块采用了节流（throttle）策略，仅在系统空闲时段执行全量扫描，同时利用文件系统事件（如 inotify / FSEvents）实现增量更新。该设计使得项目扫描成为 Hermes Agent 网页版中控台“一键式”高效管理的基石。

## 应用场景

项目扫描在以下典型场景中发挥关键作用：

1. **多项目管理**：一个开发者可能同时维护 5～10 个 GitHub 仓库，手动检查每个仓库的状态费时费力。项目扫描自动生成全局状态总览，用颜色或图标标记 Dirty 和 Stale 项目，帮助开发者快速定位需要关注的仓库。
2. **持续集成前自查**：在准备提交 Pull Request 之前，开发者可以查看项目扫描面板，确保所有相关仓库处于 Active 状态，避免因遗漏未提交的修改导致 CI 失败。
3. **自动化技能触发**：结合中控台的“重复 Prompt 闪电标记”逻辑，当某个项目持续出现 Dirty 状态超过三次时，系统会建议将其转化为一个自动化技能（Skill），例如定时执行 `git stash` 或自动生成提交信息。这一功能直接衔接了 [[Hermes Agent 网页版中控台深度分析报告]] 中提到的效率提升理念。

此外，项目扫描还与 [[健康检查]] 功能协同工作：健康检查侧重于 Hermes Agent 自身运行状态（如数据库大小、后台任务队列），而项目扫描则聚焦于开发者实际项目仓库的健康度，两者共同构成了中控台对开发环境全貌的透视能力。
