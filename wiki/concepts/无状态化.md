# 无状态化 (Statelessness)

## 概念定义
在现代多 Agent 协作架构中，“无状态化”是一种核心的设计范式，旨在将 Agent 的执行环境（即“手”）与会话上下文及记忆（即“脑”）彻底解耦。传统的 Agent 开发常将环境配置、临时文件、运行时状态与 Agent 逻辑紧密耦合，导致系统如同“宠物”一般脆弱，一旦环境污染或配置冲突，便难以维护。无状态化原则要求底层执行端（如沙盒环境）不保留任何持久化状态，任何任务的执行都应基于即时注入的上下文进行。当执行单元出现故障、代码崩溃或配置冲突时，系统无需尝试修复，而是直接销毁旧实例并拉起全新的沙盒环境。这种设计不仅极大地提升了系统的鲁棒性，还使得 Agent 能够像“牛马”一样被高效调度、并发执行，从而实现高度的可伸缩性。

## 技术细节
无状态化的实现依赖于“四层解耦模型”，通过将 Agent 的不同维度进行物理或逻辑上的分离，确保执行层的纯净：

1.  **Agent 层（静态定义）**：仅包含模型配置、系统提示词（System Prompt）等静态元数据，不包含任何运行时状态。
2.  **Session 层（会话管理）**：负责维护用户交互的序列，作为无状态执行单元的输入源。
3.  **Memory Store 层（持久化记忆）**：将 Agent 的长期记忆、历史对话记录从执行环境中剥离，存储在外部数据库或向量存储中，按需加载。
4.  **Context 层（动态注入）**：在任务执行前，将必要的上下文信息（如代码库状态、历史记忆）动态注入到沙盒环境中。

在基础设施层面，这要求计算资源从长期运行的固定容器（如传统 Docker 容器）转向毫秒级启动、动态销毁的轻量级沙盒环境（如 [[Cube Sandbox]]）。这种架构确保了每一次任务执行都是在一个“干净”的初始状态下开始的，彻底消除了环境漂移带来的不确定性。

## 应用场景
*   **高并发任务调度**：在需要同时运行大量 Agent 任务的场景中，无状态化允许系统根据负载动态扩缩容，无需担心实例间的状态同步问题。
*   **复杂代码执行与调试**：当 Agent 需要在沙盒中运行代码进行验证时，无状态化确保了前一次执行产生的临时文件或环境修改不会影响下一次任务，避免了“环境污染”。
*   **故障自愈系统**：在生产环境中，若 Agent 遇到不可预知的运行时错误，系统可直接丢弃该实例并重新初始化，无需复杂的异常恢复逻辑，降低了运维复杂度。
