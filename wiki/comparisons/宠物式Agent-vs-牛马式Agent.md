# 宠物式Agent vs 牛马式Agent

**持久绑定 vs 无状态销毁**

在多Agent协作架构的演进中，最根本的思维转变在于如何看待Agent的生命周期与管理方式。传统开发模式将Agent视为需要精心呵护的“宠物”——模型、上下文、环境三者深度绑定，每次任务失败都要手动修复环境、清理记忆，如同照顾一只娇贵的宠物。而Anthropic提出的“Managed Agents”架构则主张将Agent当作“牛马”——无状态、可随时销毁并重建，用完即弃，如同征用一头耕牛，干完活就放归，下次再临时征用另一头。

## 宠物式Agent的典型特征

- **持久绑定**：Agent的模型、系统提示词、上下文记忆、本地环境耦合在一起。一个Agent一旦启动，就持续运行，不断积累上下文。
- **环境脆弱**：若执行过程中发生错误（如代码冲突、内存溢出、环境变量污染），整个Agent可能崩溃或进入不可恢复状态。
- **记忆污染**：长期对话中，Agent会逐渐被不相关的上下文“稀释”，导致决策质量下降（即“变笨”）。
- **恢复成本高**：需要手动清理日志、重置环境、注入新的提示词，无法自动化回滚。
- **典型案例**：早期基于单一LLM的聊天机器人，每个Session对应一个独立的进程或Docker容器，但Agent配置与运行时状态混在一起。

## 牛马式Agent的核心原则

- **脑手解耦**：大脑（模型规划能力）与双手（代码执行沙盒）彻底分离。Agent本身只是静态模板，定义“我是谁”（身份、工具、系统提示词），而实际执行在独立的沙盒中运行。
- **无状态销毁**：每一个任务都在独立的沙盒（如Docker或高性能沙盒Cube Sandbox）中执行。任务结束后，沙盒立即销毁；如果任务出错，直接销毁并重新拉起干净的环境——不修复，只重建。
- **Session与Agent分离**：Agent是静态配置模板，Session是动态执行实例。同一个Agent模板可以同时启动多个独立的Session，互不干扰。Session负责记忆和文件操作，Agent只负责定义行为。正如报告所述：“Session负责我在干什么，Agent负责我是谁。”
- **上下文云端化**（Session Store）：将上下文从本地搬到云端（如Redis缓存），使得Session可以像“记忆树”一样进行Fork、回滚、克隆。编排者可以随时掌握Agent最聪明的状态，并在需要时恢复。

## 技术实现对比

| 维度 | 宠物式Agent | 牛马式Agent |
|------|-------------|-------------|
| 环境管理 | 持久容器，手动清理 | 沙盒自动创建/销毁，毫秒级启动 |
| 错误恢复 | 修复环境、回滚上下文 | 直接销毁并重建沙盒 |
| 上下文存储 | Agent本地内存 | 云端Session Store，支持分支和克隆 |
| 编排模式 | 单一Agent执行长链任务 | 编排者调度多个独立子Agent |
| 资源开销 | 长期占用GPU/CPU | 按需创建，用完即释放 |
| 典型场景 | 个人助手、一人单Agent | 企业级并发、复杂业务流程 |

## 为什么“牛马”更优？

报告引用了Anthropic的洞察：“你要扩大Agent的规模，就要解耦‘脑’和‘手’。”模型的进化速度远超代码重构速度，若将Agent与特定模型版本、环境配置深度绑定，一旦模型升级，大量包装代码就会失效。而无状态沙盒模式允许Agent模板快速适配新模型，同时通过Task Graph实现大小模型异构组合，优化Token效率。

此外，这种架构支持“蜂群”式的水平扩展。未来云端计算资源的分配将从“固定服务”转向“动态容器”：无数个根据需求瞬间创建并销毁的Agent容器组成蜂群，而非长期运行的单一守护进程。

## 实践启示

开发者在构建多Agent系统时，应停止编写复杂的、针对特定模型的“缰绳”（Harness），转而构建基于[[沙盒环境]]的解耦架构。重点关注[[Session与Agent分离]]的设计原则，以及[[Session Store]]（云端上下文存储）的实现。企业应将Agent视为[[Managed Agents]]基础设施，而非个人宠物。
