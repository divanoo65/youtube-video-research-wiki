# Hermes Agent vs OpenClaw

## 对比维度表格

| 维度 | OpenClaw | Hermes Agent |
| :--- | :--- | :--- |
| **控制核心** | Gateway（网关）：中心化后台守护进程，负责路由、调度和状态维护 | 执行循环（Execution Loop）：闭环驱动，网关、调度、工具、环境全部集成 |
| **设计逻辑** | 模块化指挥，侧重稳定性、可审计性与行为追踪 | 闭环驱动，每次执行都能反向优化决策 |
| **身份锚定** | 身份与"工作区"绑定，切换工作区即切换身份配置 | SOUL.md 全局化，跨设备保持一致的核心人格 |
| **知识管理** | 依赖人工编写技能 | 自动技能生成（[[程序化知识生成]]），通过 skill_manage 工具自主创建 |
| **记忆体系** | 未提及分层记忆 | 四层记忆体系（[[分层记忆体系]]）：核心持久层、会话历史层、扩展记忆层、程序性记忆层 |
| **安全模型** | 未明确提及安全层级 | [[五层纵深防御]]：用户授权、危险命令审批、容器隔离、MCP 凭证过滤、上下文文件扫描 |
| **迁移路径** | 无 | 提供从 OpenClaw 的无缝迁移：自动检测 ~/.openclaw 并导入记忆与技能 |
| **Cron 计划** | 未明确提及 | 内置完整定时系统，自然语言设定，每 60 秒检查一次 |

## 核心差异分析

1. **架构哲学分歧：** OpenClaw 强调**可控性和稳定性**，通过 Gateway 中心化调度实现清晰的权限边界和行为审计。Hermes Agent 则追求**自我进化和自动化**，通过执行循环将 Agent 闭环中的每次交互转化为学习机会。

2. **知识增长路径不同：** OpenClaw 的知识增长依赖开发者或用户在外部编写技能文件，属于人工驱动的扩展模式。Hermes Agent 内置了 [[程序化知识生成]] 机制，使其能在运行过程中自动抽象工作流并创建可复用技能，实现从"记住事实"到"记住方法"的进化。

3. **安全设计理念：** Hermes Agent 的 [[五层纵深防御]] 是默认开启的内建机制，反映其"能力越大责任越大"的设计理念。OpenClaw 在安全方面相对简洁，依赖 Gateway 的集中管控。

4. **身份管理差异：** OpenClaw 的多工作区绑定适合多角色隔离场景；Hermes Agent 的全局 SOUL.md 更适合个人长期使用的场景，跨设备人格一致。

## 适用场景结论

- **选 OpenClaw 的场景：** 需要严格权限隔离的多租户环境、偏好清晰可见的模块化架构、团队协作需求大于个人效率需求。
- **选 Hermes Agent 的场景：** 个人长期使用、追求自动化与自我进化能力、需要多端交叉使用（CLI/Telegram/Discord）、希望实现"一次执行、永久复用"的技能积累。
