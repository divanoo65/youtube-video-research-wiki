# 大模型 vs 小模型：逻辑汇总 vs 并发搜索效率

在 [[多Agent协作架构]] 的实践中，显露出一个核心设计原则：**根据任务性质组合使用大模型与小模型**。大模型（如 Claude Opus、GPT-4）擅长深度推理、规划与复杂文本生成，但成本高、速度慢；小模型（如 Claude Haiku、GPT-4o-mini）推理能力有限，但响应快、支持高并发。两者的角色定位可以从“逻辑汇总”与“并发搜索效率”两个维度清晰区分。

## 特性对比

| 维度 | 大模型 | 小模型 |
|------|--------|--------|
| 核心能力 | 深度推理、逻辑链跟踪、多步规划 | 快速提取、模式匹配、短文本分类 |
| Token 消耗 | 高（往往需要长上下文） | 低（通常只需短上下文） |
| 响应延迟 | 中等～高（数秒至十秒级） | 极低（百毫秒级） |
| 并发上限 | 低（受限于 API 配额与成本） | 高（可同时发起数十至数百请求） |
| 典型任务 | 汇总多个搜索结果、撰写报告、制定行动计划 | 批量并发搜索、关键词提取、状态检查 |
| 状态管理 | 需要 [[Session Store]] 保持长上下文 | 无状态或极简状态，易销毁重建 |

## 在 Managed Agents 中的分工

[[Anthropic Managed Agents]] 的编排器（Coordinator）会动态分解任务，并基于任务复杂度将子任务分发给不同规模的模型：

- **并发搜索阶段**：当需要从互联网、知识库或代码库中获取大量碎片信息时，编排器会派生出多个轻量 Worker。每个 Worker 使用小模型在独立的 [[Agent 沙盒]] 中执行快速搜索。这些沙盒是瞬时的、无状态的，任务完成后立即销毁，符合“牛马模式”的设计哲学。
- **逻辑汇总阶段**：所有搜索结果的碎片被汇集到编排器或一个专用的“主分析 Agent”中，由大模型进行综合、交叉验证、消除矛盾，并生成结构化的报告或下一步行动计划。大模型在这里消耗的 Token 虽然高，但只在关键节点出现，整体 Token 效率得以优化。

这种分工带来的直接收益是：**搜索阶段可以通过数百个并发请求在几秒内完成信息收集，而汇总阶段的深度推理又不会丢失上下文**。相比单一使用大模型做所有事情，整体响应速度提升一个数量级，成本降低 60%~80%（根据 Anthropic SDK 测试数据）。

## 技术实现细节

- **毫秒级沙盒启动**：小模型使用的沙盒需要秒级甚至毫秒级启动，例如腾讯云的 Cube Sandbox 技术，避免并发调度瓶颈。
- **上下文压缩**：大模型在汇总时要处理来自多个小模型的长篇输出，此时需要 [[Coordinator 编排器]] 协助做上下文裁剪与关键信息提取，避免 Context Window 溢出。
- **失败隔离**：小模型在高并发中可能出现错误（如 API 超时、搜索结果为空），大模型汇总时需要具备容错能力：忽略噪声、对低置信度的搜索结果标注并提示重新搜索。

## 实际示例

参考报告中的场景：一个 Agent 系统需要分析某个技术趋势，编排器启动 20 个 Worker 分别搜索不同关键词（小模型并发搜索），然后将所有片段交给一个强模型（大模型）撰写一篇综合报告。期间如果某个沙盒崩溃，直接销毁并重新拉起，不影响整体进度。

这一对比页面也为 [[FDE]]（Forward Deployed Engineer）提供了模型选型的决策框架：在部署企业级 Agent 时，优先评估任务的“计算密度”与“上下文依赖度”，而非盲目使用最强模型。

## 总结

大模型与小模型并非竞争关系，而是互补工具。理解“逻辑汇总 vs 并发搜索效率”的差异，是构建高可用、低成本 Agent 系统的关键。未来随着 [[多Agent蜂群协作]] 的普及，模型选型将成为编排器的内置能力——系统自动根据任务动态分配模型，开发者无需手动指定。
