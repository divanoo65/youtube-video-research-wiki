# Anthropic Managed Agents 架构与多 Agent 协作演进深度简报

## 执行摘要

本报告基于对 Anthropic 发布的 Managed Agents 架构及其技术实现的深入分析，探讨了人工智能 Agent 领域从“单体工具”向“企业级基础设施”和“蜂群协作”转型的核心逻辑。研究指出，当前 Agent 发展的标准答案在于**“脑手解耦”**与**“无状态化”**。通过将模型的规划决策能力（大脑）与执行代码的沙盒环境（双手）彻底分离，并引入 Session 与记忆存储的独立管理，开发者能够构建出具备高扩展性、高鲁棒性且能应对长程任务的 Agent 操作系统。这种从“养宠物”到“用牛马”的范式转变，预示着未来计算资源分配将向动态、高并发的 Agent 容器化协作方向变革。

---

## 核心主题深度分析

### 1. 架构核心：脑手解耦（Decoupling Brain and Hands）
传统 Agent 开发常面临“代码补丁过时”的困境：开发者为弥补模型记忆缺陷而编写的数千行复杂代码，往往在模型升级后沦为无用的垃圾。Anthropic 提出的破局点在于解耦：
*   **大脑（Brain）：** 负责大模型的规划与决策。
*   **双手（Hands）：** 负责执行代码的沙盒环境。
这种架构允许模型专注于指令发出，而底层的执行环境则是无状态、可随时销毁并重建的独立沙盒。

### 2. “宠物”与“牛马”的范式转变
文档通过服务器架构中的经典理论，阐述了 Agent 管理逻辑的根本变化：
*   **宠物模式（Pets）：** 早期 Agent 与本地环境、记忆深度绑定。一旦配置出错、记忆堆积或环境污染，系统就会崩溃甚至需要重装。
*   **牛马模式（Cattle）：** 在解耦架构下，Agent 被视为消耗性的执行单元。如果某个沙盒代码跑崩或配置冲突，系统会直接将其干掉并重新拉起一个干净的沙盒。这种模式消除了单一配置错误导致全盘崩溃的可能性。

### 3. Managed Agents 的四层解耦结构
通过对 Anthropic SDK 中 `beta` 目录（包含 `agents`, `environments`, `sessions`, `skills`, `memory stores`, `vaults` 等模块）的逆向分析，可以抽象出 Agent 运行的四层递进结构：

| 层级 | 组件 | 功能描述 |
| :--- | :--- | :--- |
| **第一层** | Agent (模板) | 静态定义，包括所使用的模型、系统提示词、工具集及 MCP 服务器。 |
| **第二层** | Session (实例) | 动态运行实体。同一个 Agent 模板可以派生出多个相互独立的 Session，负责记录对话线程。 |
| **第三层** | Memory Store (记忆) | 记忆不再长在 Agent 身上，而是作为独立组件挂载到 Session。支持 Fork、回滚和克隆。 |
| **第四层** | Session Store (云端存储) | 记忆从本地移至云端（如 Redis），实现上下文的跨节点恢复，使 Agent 彻底无状态化。 |

### 4. 协调者（Coordinator）与多机协作
在 Managed Agents 架构中，**协调者（Coordinator）** 是一个不直接参与执行的指挥官角色。其核心职责包括：
*   **任务拆分与派发：** 将长程任务分解，管理最多 20 个具有独立 Prompt 和工具的子 Agent。
*   **解决上下文瓶颈：** 传统单一 Agent 在长程任务中会因不断压缩上下文而丢失记忆。协调者通过调度多个独立 Agent，让每个节点从全新的上下文开始工作。
*   **模型组合优化（Token Efficiency）：** 根据任务分工，混合使用大、小模型（如 Claude Code, Codex 等），实现最佳的 Token 利用效率。

## 关键语录与背景

> **“你要扩大 Agent 的规模，就要解耦脑和手。”**
*   **背景：** 源自 Anthropic 4 月 8 日的官方博客。反映了其内部在应对模型快速进化与冗余补丁代码之间的矛盾时总结出的血泪教训。

> **“要把 Agent 当牛马去用……大脑只管发指令，底层的双手全是无状态的独立沙盒。”**
*   **背景：** 解释了为何本地运行的多 Agent（如“小龙虾”项目）容易失败。强调了基础设施健壮性来源于对执行单元的彻底隔离和随时重置的能力。

> **“Session 和 Agent 是两个完全不同的实体……Agent 负责我是谁，Session 负责我在干什么。”**
*   **背景：** 对 SDK 源码深度分析后的总结，揭示了 Managed Agents 如何通过实体分离来实现复杂的状态管理。

> **“当很多厂商现在还在不断优化如何把卖 Token 这个生意做得更好的时候，海外的厂商们已经把对 AGI 的理解以及如何落地上升到了另一个维度了。”**
*   **背景：** 作者对国内外 Agent 发展现状的反思。指出 Palantir、Anthropic 和 OpenAI 正在将 Agent 从个人工具转化为企业基础设施。

## 实践洞见与行业趋势

### 1. 动态容器化的计算变革
传统的云服务通常基于固定的长效容器。但在 Agent 蜂群架构下，计算资源的需求表现为“无数个动态 Agent 容器的瞬间创建与销毁”。这意味着计算资源的分配和计价模式将迎来变革。

### 2. 毫秒级沙盒的需求
由于 Docker 容器在启动和销毁时仍存在显著的时间耗时，未来架构将向更轻量级的沙盒演进。例如，腾讯开源的 **Cube Sandbox** 声称能实现毫秒级启动，这将是实现大规模 Agent 实时协作的关键技术支撑。

### 3. FDE（前线部署工程师）的崛起
随着 Agent 从个人工具向企业级基础设施转型，能够编排、部署和维护这些复杂 Agent 蜂群架构的专业人员（如 FDE）将成为行业的新兴需求。

### 4. 构建建议
*   **弃用单体架构：** 停止编写深度绑定的复杂 Harsness 代码，转而构建无状态的执行沙盒。
*   **引入协调者层：** 在复杂业务流中，使用专门的协调者 Agent 来管理子任务，而非寄希望于单一模型处理超长上下文。
*   **关注 Token Efficiency：** 通过任务图（Task Graph）预定义规则，利用大小模型组合完成编码、审阅、测试等不同阶段，以降低成本并提升效率。
