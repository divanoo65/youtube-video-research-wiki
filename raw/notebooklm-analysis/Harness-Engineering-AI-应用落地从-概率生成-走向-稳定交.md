# Harness Engineering：AI 应用落地从“概率生成”走向“稳定交付”的深度解析

## 执行摘要

随着 AI 技术从简单的聊天机器人向复杂的代理（Agent）系统演进，开发者面临的核心挑战已从“如何让模型听懂指令”转变为“如何在真实环境中实现稳定、高成功率的任务交付”。本简报基于对 AI 工程化最新趋势的分析，详细探讨了 **Harness Engineering（马具工程/约束工程）** 的概念及其在 AI 系统中的决定性作用。

核心结论指出：真正决定 AI 系统稳定性的往往不是模型本身，而是模型之外的运行系统。通过引入 Harness Engineering，任务成功率可从不足 70% 提升至 95% 以上。AI 工程化的重心正在从提示词（Prompt）和上下文（Context）转向对整个运行生命周期的系统化治理。

---

## AI 工程化的三个演进阶段

AI 工程化在过去两年经历了三次明显的范式转移，每一阶段都旨在解决不同维度的核心问题：

| 阶段 | 核心概念 | 解决的问题 | 本质 |
| :--- | :--- | :--- | :--- |
| **第一阶段** | **Prompt Engineering** (提示工程) | 模型有没有听懂你在说什么？ | **语言设计**：塑造局部概率空间，激发模型已有能力。 |
| **第二阶段** | **Context Engineering** (上下文工程) | 模型有没有拿到足够且正确的信息？ | **信息供给**：在合适时机输入正确信息（如 RAG、动态 SOP）。 |
| **第三阶段** | **Harness Engineering** (马具工程) | 模型在真实执行中能否持续做对？ | **系统设计**：对运行系统进行工程化，实现监控、纠偏与验收。 |

## Harness Engineering 的核心定义与架构

**Harness** 原意为“马具”或“约束装置”。在 AI 领域，**Agent = Model + Harness**，即在一个代理系统中，除模型本身外，所有决定其稳定交付的机制统称为 Harness。

一个成熟的 Harness Engineering 体系包含以下六个层面：

### 1. 信息边界层
确保模型在正确的范围内思考。包括明确角色目标、任务成功标准、信息的精简裁剪，以及结构化组织规则与状态，防止信息污染和注意力涣散。

### 2. 工具系统层
解决模型与现实世界的交互问题。
*   **工具选择**：提供适量工具，防止乱用。
*   **调用时机**：明确何时查、何时答。
*   **结果加工**：对工具返回的冗余信息进行提炼后再反馈给模型。

### 3. 执行编排层
规划模型的下一步行动。将复杂任务拆解为“理解、判断、补充、生成、检查、修正”的闭环轨道，使 AI 的工作方式接近人类经验。

### 4. 记忆与状态管理层
防止模型“失忆”。系统需独立管理三类数据：当前任务状态、对话中间结果、长期记忆与用户偏好。

### 5. 评估与观测层
解决模型“自我感觉良好”的问题。通过独立的输出验收、环境验证、自动测试日志及错误归因，让系统具备自我审视能力。

### 6. 约束、校验与恢复层
决定系统能否上线的关键。在真实环境的失败常态下（如 API 超时、格式混乱），建立“约束（准入准出）- 校验（执行前后检查）- 恢复（重试、回滚、切路径）”的确定性机制。

## 一线 AI 公司的实践案例分析

### Anthropic：应对“上下文焦虑”与“自评偏差”
*   **Context Refresh（上下文重启）**：当上下文过长导致模型丢失细节或产生收尾焦虑时，不只是压缩内容，而是更换一个干净的 Agent 承接状态，类似工程中的进程重启。
*   **角色分离机制**：将任务拆分为 **Planner（规划者）**、**Generator（执行者）** 和 **Evaluator（评估者）**。Evaluator 拥有独立环境进行真实操作验证，确保生产与验收逻辑分离。

### OpenAI：重定义工程师的角色
*   **环境设计者**：工程师不再手写代码，而是设计 Agent 运行的环境。当 Agent 出错时，通过优化环境结构（如增加能力或反馈链路）而非仅仅让模型“更努力”来解决问题。
*   **模块化指令集**：将巨大的指令文档（ent.md）改为目录索引页，按需调取子文档（设计规范、安全规则等），节省模型稀缺的上下文窗口。
*   **自动化治理系统**：将资深工程师的经验转化为系统规则（如模块依赖拦截），并将修复方案自动反馈至下一轮对话，实现代码质量的自动闭环。

## 关键引言与深度解读

> **“真正决定我们的系统能不能稳定的跑起来，往往不是模型本身，而是模型外面那套运行的系统。”**
*   **背景**：许多团队在更换最强模型、修改数百遍提示词后，任务成功率仍停留在 70% 左右。
*   **解读**：模型是概率性的，而交付需要确定性。Harness 的价值在于用确定性的工程结构包裹概率性的模型。

> **“Prompt 其实只是 Context 的一部分……而 Harness 是对整个运行系统的工程化。”**
*   **背景**：讨论三者之间的逻辑边界。
*   **解读**：三者并非替代关系，而是包含关系。从表达优化到信息优化，最终升级为系统级的治理。

> **“当 Agent 出了问题的时候，修复方案几乎从来不是要它‘更努力’一点，而是确定它缺了什么样的结构性能能力。”**
*   **背景**：OpenAI 在构建百万行代码应用时的核心心得。
*   **解读**：AI 的失败通常源于环境缺失（如缺乏反馈、工具不可用或规则不明确），而非模型智力不足。

## 行动建议与洞察

1.  **从“调优”转向“架构”**：不要过度迷信提示词的微调，应着手构建涵盖六层架构的 Harness 系统，特别是加强评估和恢复机制。
2.  **实施“按需暴露”策略**：效仿 Scale AI 或 OpenAI，采用渐进式信息展示。不要一次性塞入所有 SOP，而是建立索引，让 Agent 在触发特定任务时再获取详细细节。
3.  **建立独立的验收环境**：确保 Agent 的生成结果必须经过非生成式的、带有真实环境反馈的校验（如代码运行检查、UI 截图对比）。
4.  **关注长链路容错**：在长任务中引入“状态恢复”机制。当某一步骤失败时，系统应支持回滚到最近的稳定状态，而不是从头开始。

**总结：** AI 应用落地的核心挑战正在从“让模型看起来更聪明”转向“让模型在真实世界里稳定地工作”。Harness Engineering 是解决这一转型期痛点的关键路径。
