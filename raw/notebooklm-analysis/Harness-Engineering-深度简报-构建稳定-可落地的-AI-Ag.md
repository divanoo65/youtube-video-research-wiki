# Harness Engineering 深度简报：构建稳定、可落地的 AI Agent 系统

## 执行摘要

随着人工智能应用从简单的对话生成转向复杂的任务执行，AI 工程领域正在经历一场核心范式的转移。本简报深入探讨了近期备受关注的 **Harness Engineering（马具工程/约束工程）** 概念。

Harness Engineering 的核心观点是：决定一个 AI 系统（尤其是 Agent）能否在真实场景中稳定运行的关键，往往不在于模型本身，而在于模型外部的运行系统。简而言之，**Agent = 模型 + Harness**。当模型由于上下文过载、执行偏差或缺乏反馈而导致成功率遇到瓶颈时，Harness Engineering 提供了一套包含信息边界、工具管理、执行编排、状态管理、评估观测及故障恢复的六层架构，旨在将 AI 的“聪明才智”转化为可稳定交付的业务价值。

---

## AI 工程的三个演进阶段

AI 工程在过去两年中经历了三次明显的重心迁移，反映了开发者在不同阶段面临的核心挑战：

| 阶段 | 核心概念 | 关注点 | 解决的问题 | 本质 |
| :--- | :--- | :--- | :--- | :--- |
| **第一阶段** | **Prompt Engineering** | 指令设计 | 模型有没有听懂要求？ | 塑造局部概率空间，属于**语言设计**。 |
| **第二阶段** | **Context Engineering** | 信息供给 | 模型有没有拿到正确且足够的信息？ | 在合适时机送入正确信息，属于**信息设计**。 |
| **第三阶段** | **Harness Engineering** | 运行约束 | 模型在真实执行中能否持续做对？ | 驾驭、监督和纠偏执行过程，属于**系统设计**。 |

---

## Harness Engineering 的六层架构

一个成熟的 Harness 系统负责在模型之外构建“轨道”，确保任务不跑偏。其详细架构可拆解为以下六层：

1.  **信息边界层 (Information Boundary)**
    *   **角色与目标：** 定义“我是谁”、任务是什么、成功的标准是什么。
    *   **信息裁剪：** 避免上下文污染，确保模型只在相关的边界内思考。
    *   **结构化组织：** 将规则、当前任务、运行状态和外部证据分层清晰组织。

2.  **工具系统层 (Tool System)**
    *   **工具选择：** 解决工具太少能力不足或工具太多导致模型乱用的问题。
    *   **调用时机：** 明确何时该查，何时该答。
    *   **结果精炼：** 并非直接透传工具返回的原始数据，而是提炼与任务相关的关键信息。

3.  **执行编排层 (Execution Orchestration)**
    *   **逻辑链路：** 将搜索、总结、写代码等原子能力串联。
    *   **闭环流程：** 包含理解目标、判断信息完备性、生成输出、检查输出、修正重试的完整闭环。

4.  **记忆与状态层 (Memory and State)**
    *   **分类管理：** 明确区分当前任务状态、对话中间结果、长期记忆与用户偏好，防止系统运行越久越混乱。

5.  **评估与观测层 (Evaluation and Observation)**
    *   **独立验证：** 建立独立于生成过程的验收机制，包括环境验证、自动化测试、日志与指标监控。
    *   **自我意识：** 让系统不仅能做，还能通过反馈知道自己做得对不对。

6.  **约束、校验与恢复层 (Constraint, Validation & Recovery)**
    *   **行为边界：** 规定哪些能做，哪些不能做。
    *   **故障恢复：** 在 API 超时、格式混乱或理解偏差时，具备重试、回滚或切换路径的机制，而非从头开始。

## 行业顶尖实践案例分析

### Anthropic：应对“上下文焦虑”与建立独立审计
*   **Context Reflected（上下文重映）：** 当上下文过长导致模型丢细节或急于收尾时，不只是做压缩，而是启动一个干净的“新 Agent”进行交接，类似程序开发中的“重启进程”以解决内存泄漏。
*   **独立评估者模式：** 将任务拆分为 **Planner（规划者）**、**Generator（执行者）** 和 **Evaluator（评估者）**。评估者必须是独立的，且在真实环境中操作（如实际运行页面）而非仅仅审查代码。

### OpenAI：环境设计与按需暴露
*   **工程师角色的转变：** 工程师不再只是写代码，而是“环境设计师”。当 Agent 失败时，首要思考的是环境中缺了什么结构性能力，而非让模型“再努力一点”。
*   **目录化 Skills：** 放弃将所有规范塞进一个巨大的 `.md` 文件，而是将其变为目录索引，Agent 根据需要动态调取子文档（设计、安全规则等），解决上下文窗口稀缺问题。
*   **自动化治理：** 将资深工程师的经验（如模块分层规则）写成系统拦截规则。这些规则不仅报错，还会将修复建议反馈给 Agent 进入下一轮迭代。

## 关键引述与背景

> “真正决定我们的系统能不能稳定的跑起来，往往不是模型本身，而是模型外面那套运行的系统。”

> “Agent 等于 model 加 harness。翻译成白话：在一个 Agent 系统里，除了模型本身以外，几乎所有能决定它能否稳定交付的东西都可以算进 Harness。”

> **通俗类比：**
> *   **Prompt：** 告诉新入职员工任务流程（先宣讲，再确认需求）。
> *   **Context：** 给员工准备好客户资料、产品报价等背景文件。
> *   **Harness：** 让员工带上 Checklist，在关键节点汇报，会后核对录音，按标准验收结果。重点是有没有一套**持续观测、纠偏和验收**的机制。

## 行动洞察与落地建议

1.  **从“调优提示词”转向“设计环境”：** 当任务成功率停滞不前时，应检查系统的执行编排和故障恢复机制，而非仅仅修改 Prompt 或更换更强的模型。
2.  **实施分层信息策略：** 针对上下文窗口这一稀缺资源，采用“进阶式”暴露信息的思路。只在模型需要触发特定能力时，才动态加载相关的详细 SOP 和参数定义。
3.  **构建独立的评估闭环：** 必须建立“生产与验收分离”的机制。确保评估者具备在真实环境中运行和验证结果的能力，形成“生成—检查—修复”的有效循环。
4.  **容错是系统的基本属性：** 在长链路任务中，失败是常态。Harness 系统必须具备状态保存和路径回滚能力，确保系统在部分步骤失败后能够拉回正轨并稳定交付。
