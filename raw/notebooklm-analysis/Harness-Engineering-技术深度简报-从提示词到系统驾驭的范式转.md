# Harness Engineering 技术深度简报：从提示词到系统驾驭的范式转移

## 执行摘要

随着人工智能从“问答机器人”向“任务执行智能体（Agent）”演进，AI 工程化的核心挑战已从“让模型变聪明”转向“让模型在真实世界里稳定工作”。**Harness Engineering（马具工程/约制工程）** 正在成为决定 AI 应用能否落地的关键。

本报告基于对最新 AI 工程实践的分析，探讨了从提示词工程（Prompt Engineering）、上下文工程（Context Engineering）到 Harness Engineering 的三阶段演进过程。Harness Engineering 的核心思想是：**真正决定系统稳定性的往往不是模型本身，而是模型外部的运行系统（即 Harness）**。通过构建包含六层架构的 Harness 系统，开发者可以将 Agent 的任务成功率从较低水平（如 70%）提升至准生产级别（如 95% 以上）。

---

## AI 工程化的演进路径

过去两年中，AI 工程经历了三次显著的重心迁移，分别对应 AI 系统发展的三个阶段性核心问题：

| 阶段 | 核心概念 | 关注点 | 解决的问题 | 本质 |
| :--- | :--- | :--- | :--- | :--- |
| **第一阶段** | **Prompt Engineering** | 语言设计 | 模型有没有听懂指令？ | 塑造局部概率空间，激发模型已有能力。 |
| **第二阶段** | **Context Engineering** | 信息供给 | 模型有没有拿到正确的信息？ | 在合适时机将正确信息（RAG、工具返回等）送入模型。 |
| **第三阶段** | **Harness Engineering** | 系统驾驭 | 模型在真实执行中能否持续做对？ | 对整个运行系统的工程化，负责监督、约束和纠偏。 |

---

## Harness Engineering 的六层架构分析

一个成熟的 Harness Engineering 系统由以下六个层次构成，旨在为模型提供稳定的“轨道”：

### 1. 信息边界层 (Information Boundary)
确保模型在正确的边界内思考。包括：
*   **角色目标定义：** 明确“我是谁”、“任务是什么”、“成功标准是什么”。
*   **信息裁剪与选择：** 上下文并非越多越好，需保持高度相关性。
*   **结构化组织：** 分层组织规则、当前任务状态与外部证据，防止信息污染导致模型遗漏约束。

### 2. 工具系统层 (Tool System)
将模型从文本预测器转变为行动者。
*   **工具选择：** 解决工具太少能力不足、工具太多模型乱用的平衡问题。
*   **调用时机：** 明确何时该查资料，何时不该乱查。
*   **结果炼制：** 搜集到的原始结果需经过提炼、筛选后再反馈给模型，保持任务相关性。

### 3. 执行编排层 (Execution Orchestration)
解决“下一步该做什么”的问题，将离散步骤串联为逻辑链条：
*   **闭环流程：** 理解目标 -> 判断信息 -> 补全信息 -> 执行 -> 分析结果 -> 检查输出 -> 修正/重试。

### 4. 记忆与状态管理层 (Memory & State)
防止 Agent 出现“失忆”现象，需区分并管理三类数据：
*   **当前任务状态：** 正在进行的步骤。
*   **中间结果：** 会话过程中的阶段性结论。
*   **长期记忆：** 用户偏好等持久化信息。

### 5. 评估与观测层 (Evaluation & Observation)
建立独立于生成的验证机制：
*   **自省能力：** 判定输出是否合规、环境是否验证通过。
*   **工程监控：** 包含自动测试、日志指标归因，让系统感知自己“有没有做对”。

### 6. 约束、校验与恢复层 (Constraints, Validation & Recovery)
决定系统能否上线的关键，将“失败”视为常态进行处理：
*   **约束：** 明确行为红线。
*   **校验：** 输出前后的严格检查。
*   **恢复机制：** 针对 API 超时、格式混乱等错误，实施重试、切换路径或回滚。

## 行业领先实践案例

### Anthropic：解决上下文焦虑与评分偏差
*   **上下文反射 (Context Reflection)：** 发现模型在上下文快满时会产生“焦虑”并急于收尾。Anthropic 不仅是压缩上下文，而是直接更换一个新的、干净的 Agent 承接状态，类似工程中的“进程重启”。
*   **角色分离机制：** 采用 **Planner（规划者）**、**Generator（执行者）** 和 **Evaluator（评估者）** 分离的模式。Evaluator 必须具备独立性，能真实操作页面、检查交互结果，而非仅仅审阅代码。

### OpenAI：环境设计与按需暴露
*   **从代码编写转向环境设计：** 工程师的职责变为：拆解任务、识别环境缺失能力、建立反馈链。
*   **索引化指令 (Agent.md)：** 放弃将所有规则塞进单一庞大文档，而是将其改造成“目录页”，仅保留核心索引，详细规范（设计文档、质量评分等）作为子文档通过“按需暴露”的方式提供给模型。
*   **自驱动治理系统：** 将资深工程师的经验（如模块分层规则）写成系统规则，拦截错误并同时反馈修改方案，形成自动治理循环。

### Scale AI：Agent Skills 实践
*   **渐进式暴露：** 初始仅给模型展示最少量的原型，只有当特定能力被触发时，才动态加载相关的 SOP、参数定义和脚本，解决注意力涣散问题。

## 重要语录与上下文

> **“Agent = Model + Harness (Harness = Agent - Model)”**
> *   **背景：** 一线 AI 工程师给出的典型定义，强调在 Agent 系统中，除模型外的所有决定稳定交付的部分皆属于 Harness。

> **“提示词工程的本质不是命令模型，而是塑造一个局部的概率空间。”**
> *   **背景：** 解释为什么 PE 有效：模型对上下文敏感，通过设定身份和范式来引导其生成的概率。

> **“当 Agent 出问题的时候，修复方案几乎从来不是‘要更努力一点’，而是确定它缺了什么样的结构性能能。”**
> *   **背景：** OpenAI 的工程思维，强调通过优化 Harness（环境和规则）而非单纯依赖模型智力来解决问题。

## 行动建议与洞察

1.  **从“调优”转向“架构”：** 如果 Agent 成功率遇到瓶颈，不要仅局限于修改提示词或更换模型，应重点审查任务拆解方式、状态管理和校验逻辑。
2.  **建立独立的评估平面：** 生产环境下的验证必须脱离生成者本身。构建能够模拟真实用户操作、检查日志和监控的独立验证 Agent。
3.  **优化上下文密度：** 遵循“按需暴露”原则。将冗长的文档和规则索引化，避免信息过载导致的注意力涣散或“上下文焦虑”。
4.  **构建容错机制：** 接受模型会跑偏或失败的现实。在 Harness 中内置自动重试、路径切换和状态回滚功能，确保系统具备自愈能力。
