# 多 Agent 协作架构深度简报：从“宠物”到“牛马”的范式演进

## 执行摘要

本报告基于对 Anthropic 托管 Agent（Managed Agents）架构的深度分析与实践复刻，探讨了多 Agent 协作领域的最新技术趋势。核心观点指出，当前的 Agent 开发已进入“编排+蜂群”时代。通过将大模型的“大脑”（规划决策能力）与“双手”（执行环境沙盒）彻底解耦，开发者能够构建出具备高鲁棒性、可大规模扩展且无状态的 Agent 集群。这种架构转变标志着 Agent 从单一的个人工具演进为企业级的底层基础设施。

---

## 核心主题深度分析

### 1. 架构范式的根本转变：从“宠物”到“牛马”
传统 Agent 开发模式中，开发者倾向于编写数千行复杂的修补代码（Harness）来弥补模型记忆缺陷，这导致 Agent 与本地环境深度绑定。
*   **“宠物”模式（旧）**：Agent 拥有状态且环境固定。一旦配置出错、环境污染或记忆过载，系统便会崩溃，必须重装或手动修复。
*   **“牛马”模式（新）**：Agent 被视为无状态的消耗品。底层的“双手”是独立的沙盒，一旦运行报错或冲突，直接销毁并重新拉起干净的沙盒继续任务。

### 2. 四层解耦架构模型
通过对 Anthropic SDK 的逆向分析，该架构可以抽象为层层递进的四层结构：

| 维度 | 名称 | 核心职能 | 特性 |
| :--- | :--- | :--- | :--- |
| **第一层** | **Agent (静态模板)** | 定义“我是谁”。包含模型、系统提示词、工具集、MCP服务器。 | 静态定义，可复用。 |
| **第二层** | **Session (动态执行)** | 定义“我在干什么”。承载对话线程、记忆挂载和临时文件。 | 独立运行，支持并发。 |
| **第三层** | **Session Store (记忆管理)** | 记忆与环境的解耦。支持记忆树的 Fork、回滚、克隆。 | 提升复杂任务的容错率。 |
| **第四层** | **Managed Infra (托管设施)** | 实现 Agent 的彻底无状态化。 | 随时销毁，秒级重建。 |

### 3. 编排者（Coordinator）与任务图
在多 Agent 协作中，核心不再是单一 Agent 的强度，而是编排能力。
*   **协调员职能**：一个不直接参与执行的 Agent，负责拆解任务、派发工作给最多 20 个具备不同 Skill 的 Agent。
*   **异步协作与任务图**：通过预定义的任务图（Task Graph），实现如“编码 -> 评审 -> 测试”的并发流水线。
*   **Token 效率优化**：根据任务阶段的不同，组合使用大模型（处理复杂决策）和小模型（处理简单任务），实现最佳的 Token Efficiency。

### 4. 基础设施的演进：沙盒化与 Agent OS
随着 Agent 数量的激增，传统的 Docker 容器因启动速度限制可能面临瓶颈。
*   **毫秒级启动**：如腾讯开源的 Cube Sandbox 等技术正在解决沙盒的启动效率问题。
*   **Agent 操作系统**：云端正在形成一种新型的“Agent 操作系统”，计算资源的分配和计价模式将从长期固定的容器转向动态创建、即用即走的 Agent 蜂群。

## 关键引言及上下文分析

1.  **“你要扩大 Agent 的规模，就要解耦脑和手。”**
    *   **上下文**：源自 Anthropic 官方博客。强调了在模型快速进化背景下，硬编码修补程序的低效性。解耦是实现 Agent 规模化运行的前提。

2.  **“模型的进化速度，永远比你重构代码的速度快。”**
    *   **上下文**：作者反思 Anthropic 工程师曾为弥补旧模型缺陷编写大量代码，但新模型发布后，这些代码瞬间成为“毫无价值的垃圾”。这警示开发者应关注架构而非补丁。

3.  **“Agent 负责我是谁，Session 负责我在干什么。”**
    *   **上下文**：用于解释 Agent 模板与动态会话之间的解耦关系。同一个 Agent 模板可以同时跑 10 个相互独立的 Session。

4.  **“海外厂商们已经把对 AGI 的理解以及如何落地，上升到了另一个维度了。”**
    *   **上下文**：作者通过对比 Palantir 的 AIP 系统、Anthropic 的 Managed Agents 与国内仍关注“卖 Token”的现状，呼吁国内开发者反思技术路径。

## 行动指南与洞察

### 1. 优先实施解耦策略
*   **行动建议**：停止编写依赖于特定模型缺陷的复杂 Harness 代码。
*   **目标**：构建无状态的执行节点，将模型调用逻辑与代码执行沙盒彻底分离。

### 2. 构建基于 Session Store 的记忆管理
*   **行动建议**：将 Context（上下文）从本地迁移至云端缓存（如 Redis）。
*   **目标**：实现对话状态的可追溯、可回滚和可克隆，解决长程任务中的上下文压缩和信息丢失问题。

### 3. 优化 Token Efficiency
*   **行动建议**：在编排层引入多模型分工。
*   **目标**：在任务图的不同阶段，将简单任务（如初步整理、格式转换）交给小模型，将核心逻辑交给高性能大模型，以降低成本并提升速度。

### 4. 关注新型基础设施建设
*   **行动建议**：调研并评估高性能沙盒环境（如轻量化容器、毫秒级启动沙盒）。
*   **目标**：为未来企业级 Agent 蜂群的高并发需求做好技术储备，将 Agent 从个人辅助工具提升为企业基础设施。
